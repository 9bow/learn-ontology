## OWA vs CWA: 두 세계관 비교

```text
┌──────────────────────────────┐  ┌──────────────────────────────┐
│   CWA (닫힌 세계 가정)        │  │   OWA (열린 세계 가정)        │
│   Closed World Assumption    │  │   Open World Assumption      │
├──────────────────────────────┤  ├──────────────────────────────┤
│                              │  │                              │
│  DB에 없으면 → 거짓(false)   │  │  명시 안 됐으면 → 알 수 없음  │
│                              │  │  (unknown, not false)        │
│  이영희의 자격증 정보 없음   │  │  이영희의 자격증 정보 없음    │
│         ↓                    │  │         ↓                    │
│  "이영희는 자격증 없다" ✅   │  │  "자격증 여부 모른다" ⚠️     │
│                              │  │  (미래에 추가될 수 있음)      │
├──────────────────────────────┤  ├──────────────────────────────┤
│  부정: 암묵적                │  │  부정: 명시적 선언 필요       │
│  (Negation as Failure)       │  │  owl:complementOf 등         │
├──────────────────────────────┤  ├──────────────────────────────┤
│  적합: 트랜잭션, 운영 시스템 │  │  적합: 지식 통합, 링크드 데이터│
│  표준: SQL, 관계형 DB        │  │  표준: OWL 2                 │
└──────────────────────────────┘  └──────────────────────────────┘

          SHACL ◀── OWA 환경에서 CWA 스타일 검증을 제공
```

---

## 두 가지 세계관

온톨로지를 처음 배우는 데이터베이스 엔지니어가 가장 많이 혼란스러워하는 부분이 있습니다. "데이터베이스에 없으면 거짓인데, 온톨로지에서는 왜 '알 수 없음'이 되나요?" 이 질문이 바로 **열린 세계 가정(Open World Assumption, OWA)** 과 **닫힌 세계 가정(Closed World Assumption, CWA)** 의 차이를 건드립니다.

이 두 가정은 단순한 기술적 차이가 아닙니다. 지식을 표현하는 철학적 입장의 차이입니다. 그리고 이 차이를 이해하지 못하면, OWL로 작성한 온톨로지가 "왜 이 추론이 안 나오나?"며 당황하게 됩니다.

---

## 닫힌 세계 가정 (CWA)

**CWA** 는 "데이터베이스에 명시되지 않은 것은 거짓(false)이다"라는 가정입니다. 관계형 데이터베이스의 기본 가정입니다.

예시를 봅시다. 직원 데이터베이스에 다음 테이블이 있습니다.

```sql
CREATE TABLE Employee (
  name     VARCHAR(100),
  dept     VARCHAR(50),
  has_cert BOOLEAN
);

INSERT INTO Employee VALUES ('김철수', '개발팀', TRUE);
INSERT INTO Employee VALUES ('이영희', '마케팅팀', FALSE);
```

질의: "인증서가 없는 직원은 누구인가?"

```sql
SELECT name FROM Employee WHERE has_cert = FALSE OR has_cert IS NULL;
```

결과: 이영희. 김철수는 제외. 이영희의 `has_cert`가 `FALSE`이거나, 다른 직원의 레코드가 없으면 "그 직원은 인증서가 없다"고 결론 내립니다. **기록되지 않은 사실 = 거짓** 이 CWA입니다.

이것은 트랜잭션 시스템에서 합리적입니다. 병원 예약 시스템에서 "9시에 예약이 없다"면 그 시간은 비어 있는 것으로 처리해야 합니다. "알 수 없음"으로 처리하면 예약 로직이 동작하지 않습니다.

---

## 열린 세계 가정 (OWA)

**OWA** 는 "명시적으로 거짓이라고 선언하지 않으면, 참일 수도 있다(또는 알 수 없다)"라는 가정입니다. 온톨로지와 OWL의 기본 가정입니다.

같은 상황을 OWL로 표현해봅니다.

```turtle
:Employee rdf:type owl:Class .
:hasCertification rdf:type owl:DatatypeProperty ;
                  rdfs:domain :Employee ;
                  rdfs:range xsd:boolean .

:김철수 rdf:type :Employee ;
        :hasCertification true .

:이영희 rdf:type :Employee .
# :이영희의 hasCertification이 없음
```

OWA 환경에서 추론 엔진은 `:이영희`에 대해 "인증서가 없다"고 결론 내리지 **않습니다.** 단지 "인증서 보유 여부를 모른다"고 판단합니다. 미래에 `:이영희 :hasCertification true`라는 트리플이 추가될 수 있기 때문입니다.

이것은 **분산된 지식 환경**에서 합리적입니다. Wikidata에 어떤 인물의 출생지 정보가 없다면, 그 인물이 출생지가 없는 것이 아니라 단지 아직 입력되지 않은 것입니다.

## OWA가 만드는 결정적 차이: 세 가지 시나리오

### 시나리오 1: 자격 검사

**CWA(DB):** "자격증 데이터가 없으면 자격 없음 → 접근 거부"

**OWA(OWL):** "자격증 데이터가 없으면 알 수 없음 → 추론 엔진은 '자격 없음'을 도출하지 않음"

```turtle
# OWA에서는 이 쿼리가 예상대로 동작하지 않을 수 있음
# "인증서가 없는 직원"을 SPARQL로 찾으려면 명시적 negation 필요

SELECT ?emp WHERE {
  ?emp rdf:type :Employee .
  FILTER NOT EXISTS { ?emp :hasCertification true }
}
# 이 쿼리는 "인증서가 true로 기록되지 않은 직원"을 반환
# ≠ "인증서가 없는 직원"
```

### 시나리오 2: owl:FunctionalProperty와 OWA

```turtle
:hasSSN rdf:type owl:DatatypeProperty ,
                 owl:FunctionalProperty .

:홍길동 :hasSSN "111111-1111111" .
:홍길동 :hasSSN "222222-2222222" .
```

CWA 환경(DB)이라면 이것은 데이터 오류입니다. OWA 환경(OWL)에서 추론 엔진은 다르게 반응합니다. `FunctionalProperty`에 의해 두 값은 반드시 같은 값이어야 하므로, 추론 엔진은 `"111111-1111111" owl:sameAs "222222-2222222"`를 도출하려 합니다. 리터럴은 `sameAs`가 불가능하므로 **불일관성(inconsistency)** 이 발생합니다.

### 시나리오 3: 부정 정보 표현

OWL에서 "A는 B가 아니다"를 명시하려면 의식적으로 선언해야 합니다.

```turtle
# "김철수는 간호사가 아니다"를 명시
:김철수 rdf:type [
    rdf:type owl:Class ;
    owl:complementOf :Nurse
] .

# 또는 개체 간 비동일성
:DrKim owl:differentFrom :NurseChoi .
```

이것을 잊으면, OWA 환경에서 `:DrKim`이 `:Nurse`임을 부정할 방법이 없습니다.

## OWA와 CWA의 비교표

| 항목 | CWA (DB/SQL) | OWA (OWL) |
|------|-------------|----------|
| 기본 입장 | "없으면 거짓" | "없으면 알 수 없음" |
| 부정 처리 | 암묵적 (Negation as Failure) | 명시적 선언 필요 |
| 추론 방향 | 현재 데이터 내 완결 | 미래 지식 추가 가능 |
| 적합한 환경 | 트랜잭션, 운영 시스템 | 지식 통합, 링크드 데이터 |
| 일관성 검사 | FK 제약, 트리거 | 추론 엔진 (HermiT 등) |
| 표준 | SQL | OWL 2 |

## SHACL: OWA 환경의 CWA 스타일 검증

OWA의 추론 능력은 강력하지만, 실무에서는 데이터 품질 검증이 필요합니다. "인증서 정보가 반드시 있어야 한다"는 운영 요구사항은 OWL만으로 표현하기 어렵습니다.

이 문제를 해결하는 것이 **SHACL(Shapes Constraint Language)** 입니다. SHACL은 OWA 환경에서 CWA 스타일의 제약을 적용합니다. 추론이 아니라 **검증(validation)** 이 목적입니다.

```turtle
# SHACL 예시: 직원은 반드시 부서 정보를 가져야 한다
@prefix sh: <http://www.w3.org/ns/shacl#> .

:EmployeeShape rdf:type sh:NodeShape ;
               sh:targetClass :Employee ;
               sh:property [
                   sh:path :hasDepartment ;
                   sh:minCount 1 ;
                   sh:message "직원에게는 반드시 부서가 있어야 합니다"@ko
               ] .
```

| 역할 | OWL | SHACL |
|------|-----|-------|
| 목적 | 지식 표현 및 추론 | 데이터 검증 |
| 세계 가정 | OWA | CWA 스타일 |
| 위반 시 | 불일관성 또는 무시 | 검증 보고서 생성 |
| 표준 | W3C OWL 2 | W3C SHACL |

SHACL은 섹션 03의 챕터 3-4에서 상세히 다룹니다.

## 실무 설계 함의

OWA를 이해하면 온톨로지 설계가 달라집니다.

**필수 관계는 Restriction으로 표현하라** — `someValuesFrom`으로 "적어도 하나의 처방이 있어야 함"을 표현하면, 추론 엔진이 누락을 탐지합니다. 단, 이것도 "해당 정보가 미지(unknown)"인 상황에서는 불일관성을 일으키지 않습니다.

**검증이 목적이라면 SHACL을 병행하라** — 운영 데이터 파이프라인에서 필수 필드, 형식 검증, 값 범위 검사는 SHACL이 더 적합합니다.

**부정 정보를 적극적으로 표현하라** — "이 약물은 소아에게 금기다" 같은 금기 정보는 `owl:complementOf`, `owl:disjointWith`, 또는 명시적 부정 속성으로 표현해야 합니다.

다음 챕터에서는 추론의 세 가지 핵심 유형 — Subsumption, Classification, Consistency — 을 살펴봅니다.
