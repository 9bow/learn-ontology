## 거버넌스 없는 온톨로지의 운명

거버넌스 체계 없이 구축된 지식 그래프는 예측 가능한 방식으로 무너집니다. 처음에는 소수의 전문가가 온톨로지를 설계하지만, 조직이 성장하면서 여러 팀이 각자 필요한 클래스를 추가합니다. 검토 없이 추가된 클래스들이 서로 중복되거나 충돌하고, 몇 년 후에는 아무도 전체 구조를 이해하지 못하는 상태가 됩니다. 이것을 **온톨로지 스프롤(Ontology Sprawl)** 이라고 합니다.

거버넌스는 이를 방지하는 제도적 장치입니다.

---

## 거버넌스 위원회 구성

**온톨로지 거버넌스 위원회(Ontology Governance Board, OGB)** 는 다음 역할을 담당합니다.

| 역할 | 책임 | 구성원 예시 |
|------|------|---------|
| 의장 | 최종 결정권, 이해관계 조정 | 수석 데이터 아키텍트 |
| 도메인 대표 | 도메인별 요구사항 대변 | 재무팀 데이터 리드, HR 데이터 리드 |
| 온톨로지 엔지니어 | 기술적 타당성 검토 | 온톨로지 설계 담당자 |
| 보안 담당 | 접근 제어 정책 검토 | 정보보안팀 |
| 사용자 대표 | 실사용 관점 검토 | 분석팀 대표 |

**의사결정 프로세스:**

1. **변경 요청(Change Request) 제출:** 누구나 제출 가능. 클래스 추가, 속성 변경, 기존 개념 폐기 등을 포함합니다.
2. **기술 검토:** 온톨로지 엔지니어가 기술적 타당성과 기존 온톨로지와의 충돌을 검토합니다.
3. **도메인 검토:** 관련 도메인 대표가 비즈니스 요구사항과의 부합 여부를 검토합니다.
4. **승인 또는 반려:** OGB 투표로 결정합니다.
5. **배포 및 문서화:** 승인된 변경은 버전 관리 시스템에 커밋하고 변경 이력을 기록합니다.

## 온톨로지 버전 관리

OWL 2는 온톨로지 버전 관리를 위한 공식 메커니즘을 제공합니다.

```turtle
<https://example.com/ontology/finance>
    a owl:Ontology ;
    owl:versionIRI <https://example.com/ontology/finance/1.2.0> ;
    owl:priorVersion <https://example.com/ontology/finance/1.1.0> ;
    owl:backwardCompatibleWith <https://example.com/ontology/finance/1.1.0> ;
    rdfs:comment "FIBO 2024Q1 업데이트 반영, ESG 클래스 추가"@ko ;
    dct:modified "2024-04-01"^^xsd:date .
```

버전 번호 체계(Semantic Versioning):
- **Major(X.0.0):** 하위 호환성 깨지는 변경 (클래스 삭제, URI 변경)
- **Minor(0.X.0):** 하위 호환 추가 (새 클래스, 새 속성)
- **Patch(0.0.X):** 레이블·주석 수정, 버그 수정

## Named Graph 기반 접근 제어

단일 트리플 스토어에서 여러 팀의 데이터를 함께 관리할 때, 접근 제어가 필수입니다. **Named Graph**는 트리플 집합에 URI를 부여해 단위별 관리를 가능하게 합니다.

```turtle
# 공개 데이터 그래프
GRAPH <https://example.com/graph/public> {
    :ProductA :hasName "제품 A" .
}

# 내부 가격 정보 그래프 (재무팀만 접근)
GRAPH <https://example.com/graph/pricing> {
    :ProductA :hasWholesalePrice "50000"^^xsd:decimal .
}

# 고객 데이터 그래프 (개인정보 처리 동의 필요)
GRAPH <https://example.com/graph/customer-pii> {
    :Customer123 :hasEmail "user@example.com" .
}
```

그래프 수준 접근 제어는 트리플 스토어 제품마다 구현 방식이 다릅니다.

| 트리플 스토어 | 접근 제어 방식 |
|------------|-------------|
| Stardog | RBAC + Named Graph 레벨 권한 |
| GraphDB | Roles + Graph-level permissions |
| Apache Jena Fuseki | 기본 미지원, 프록시 레이어 필요 |
| Amazon Neptune | IAM + Resource-based policy |

## 데이터 계보 추적

거버넌스의 핵심은 **"이 데이터는 어디서 왔는가"를 추적하는 능력**입니다. PROV-O(W3C Provenance Ontology)는 이를 위한 표준 어휘입니다.

```turtle
# 특정 트리플의 출처를 PROV-O로 기록
GRAPH <https://example.com/graph/provenance> {
    _:extraction1 a prov:Activity ;
        prov:startedAtTime "2024-03-15T09:00:00Z"^^xsd:dateTime ;
        prov:wasAssociatedWith :ETLPipeline_v2 ;
        prov:used <https://source.example.com/api/companies> .

    <https://example.com/graph/companies>
        prov:wasGeneratedBy _:extraction1 ;
        prov:wasDerivedFrom <https://source.example.com/api/companies> .
}
```

이 계보 정보를 유지하면 소스 데이터가 변경됐을 때 영향을 받는 트리플을 파악하고, 규제 감사(Audit) 시 데이터의 출처를 설명할 수 있습니다.

## 보안 설계 원칙

지식 그래프는 조직의 핵심 데이터를 통합하므로 보안 설계가 중요합니다.

**최소 권한 원칙:** 각 사용자와 서비스는 필요한 최소한의 그래프에만 읽기/쓰기 권한을 가집니다.

**암호화:** 전송 중 데이터(TLS 1.3), 저장 중 데이터(at-rest encryption)를 모두 암호화합니다. SPARQL 엔드포인트는 HTTPS로만 노출합니다.

**감사 로그:** 누가 언제 어떤 쿼리를 실행했는지 로그를 기록합니다. 특히 개인정보가 포함된 그래프에 대한 접근은 별도 감사 로그를 유지합니다.

**인젝션 방어:** SPARQL 인젝션 공격을 방어하기 위해 파라미터화된 쿼리를 사용하고, 사용자 입력을 직접 SPARQL 문자열에 삽입하지 않습니다.

```python
# 잘못된 방식: 문자열 직접 삽입 (SPARQL 인젝션 위험)
query = f"SELECT * WHERE {{ ?s :name '{user_input}' }}"

# 올바른 방식: 파라미터화 (라이브러리별 방법 상이)
query = "SELECT * WHERE { ?s :name ?name }"
result = graph.query(query, initBindings={"name": Literal(user_input)})
```

## 정리

데이터 거버넌스는 지식 그래프의 장기적 건강을 유지하는 제도적 기반입니다. 거버넌스 위원회, 버전 관리, Named Graph 기반 접근 제어, PROV-O 계보 추적을 갖추면 조직이 성장하면서도 온톨로지의 무결성을 유지할 수 있습니다. 다음 챕터에서는 조직 변화 관리와 도메인 전문가 협업 방법을 살펴봅니다.
