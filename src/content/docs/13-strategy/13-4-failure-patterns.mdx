## 실패는 흔하다

온톨로지와 지식 그래프 프로젝트의 실패율은 높습니다. 시맨틱 웹의 1990년대 후반~2000년대 초반 1차 붐은 과장된 기대와 구현 복잡성의 충돌로 실망기를 맞았습니다. 링크드 데이터(Linked Data) 운동의 2000년대 후반 붐도 비슷한 패턴을 겪었습니다. 지식 그래프는 현재 새로운 붐을 맞고 있습니다.

과거의 실패 패턴을 이해하면 같은 함정을 피할 수 있습니다.

---

## Gartner Hype Cycle과 기술 성숙도

**Gartner Hype Cycle**은 신기술이 거치는 5단계 심리적 사이클을 묘사합니다.

| 단계 | 특징 | 지식 그래프 관점 |
|------|------|--------------|
| 기술 촉발 | 초기 발표, 언론 관심 | LLM+KG 결합 기사 급증 |
| 부풀려진 기대의 정점 | "이게 모든 것을 해결한다" | "지식 그래프로 데이터 문제 해결" |
| 환멸의 골짜기 | 과장이 드러남, 프로젝트 취소 | 시맨틱 웹 1차 붐의 실망기 |
| 계몽의 비탈 | 실제 적용 사례 축적 | 구글 KG, 의료 SNOMED CT |
| 생산성의 안정기 | 주류 채택 | 현재 기업 KG의 위치 |

지식 그래프는 현재 계몽의 비탈에서 생산성 안정기로 이동 중인 것으로 평가됩니다. LLM과의 결합(GraphRAG 등)은 새로운 기대 정점을 만들고 있습니다. 이 사이클을 인식하면 과도한 기대를 조절하고 현실적 계획을 세울 수 있습니다.

## 실패 패턴 1: Ontology Theatre

**Ontology Theatre(온톨로지 극장)** 는 온톨로지가 실제로 사용되지 않으면서 존재한다는 착각을 만드는 상태입니다. 다음 증상으로 진단할 수 있습니다.

- 온톨로지가 설계됐지만 실제 데이터가 매핑되지 않았다
- SPARQL 엔드포인트가 있지만 아무도 쿼리하지 않는다
- 문서는 있지만 누가 읽는지 모른다
- 온톨로지를 "완성"했다고 발표했지만 비즈니스 문제가 해결되지 않았다

**원인:** 기술적 결과물(온톨로지 파일 생성)이 목표가 되고, 비즈니스 가치 창출이 뒤로 밀립니다.

**대응:** 온톨로지 완성이 아니라 역량 질문 해결을 마일스톤으로 설정합니다. "CQ-007번 질문에 SPARQL로 정확히 답할 수 있게 됐다"가 진짜 완료 기준입니다.

---

## 실패 패턴 2: 범위 크리프(Scope Creep)

"공급망 온톨로지를 만들자"에서 시작해 "그럼 제품 온톨로지도 필요하고, 고객 온톨로지도 연결해야 하고, 재무 데이터도 붙이자"로 번져가는 패턴입니다.

**증상:**
- 초기 범위의 3배 이상으로 늘어났다
- 완료 예상 시점이 계속 연기된다
- 팀이 지쳐있다

**원인:** 온톨로지의 연결성은 장점이자 함정입니다. 모든 개념이 연결되어 있어 경계를 긋기 어렵습니다.

**대응:** "이 온톨로지가 답해야 하는 역량 질문 목록"을 경계로 사용합니다. 목록에 없는 것은 다음 버전으로 미룹니다. 온톨로지는 버전 1.0이 완벽할 필요가 없습니다.

---

## 실패 패턴 3: 완벽주의 함정

"이 개념이 정말 클래스여야 하는가, 속성이어야 하는가?"를 몇 달씩 논의하다가 아무것도 구현하지 못하는 상태입니다.

**증상:**
- 철학적 토론이 실용적 설계 결정보다 오래 걸린다
- "완벽한 온톨로지가 나오면 시작하자"는 분위기
- POC가 6개월이 지나도 시작되지 않는다

**원인:** 온톨로지 설계는 정답이 없는 결정의 연속입니다. 완벽을 추구하면 마비됩니다.

**대응:** "지금 결정으로 충분히 앞으로 나갈 수 있는가?"를 기준으로 사용합니다. 틀린 결정은 나중에 수정할 수 있지만, 결정하지 않으면 아무것도 진행되지 않습니다. 온톨로지는 가설의 집합이며, 데이터와 사용을 통해 검증됩니다.

---

## 실패 패턴 4: 도구 우선 접근

"Neo4j를 사고 나서 뭘 할지 생각하자"는 패턴입니다. 도구가 솔루션을 결정하면 문제가 도구에 맞게 재정의됩니다.

**증상:**
- 도구 선택이 요구사항 분석보다 먼저 이루어졌다
- "우리는 그래프 DB를 사서 모든 데이터를 넣었는데 아무도 안 쓴다"
- 도구의 제약이 온톨로지 설계를 왜곡한다

**대응:** 역량 질문 → 온톨로지 설계 → 도구 선택 순서를 지킵니다. 도구는 온톨로지를 구현하는 수단이지 온톨로지가 도구에 종속되어서는 안 됩니다.

---

## 실패 패턴 5: 단일 실패 지점(Bus Factor 1)

온톨로지를 한 사람만 이해하고 유지보수하는 상태입니다. 그 사람이 떠나면 프로젝트가 멈춥니다.

**버스 팩터(Bus Factor):** 몇 명이 버스에 치이면 프로젝트가 멈추는가. 1이면 매우 취약합니다.

**대응:**
- 온톨로지 문서화를 강제화합니다
- 페어 온톨로지 리뷰(두 명이 함께 검토)를 프로세스화합니다
- 신규 팀원에게 온톨로지 수정 작업을 경험하게 합니다

---

## 실패 패턴 6: 데이터 품질 과소평가

"깨끗한 데이터로 시작하면 됩니다"는 가정이 틀리는 경우입니다. 실제 프로덕션 데이터는 예상보다 훨씬 지저분합니다.

**실제 데이터에서 흔히 발견되는 문제:**
- 같은 법인이 10가지 다른 이름으로 입력됨
- 필수 필드가 20%의 레코드에서 누락
- 날짜 형식이 "2024-01-15", "01/15/2024", "2024년 1월 15일"로 혼재
- 코드 값이 문서화되지 않아 의미 불명

**대응:** SHACL 검증을 데이터 수집 초기 단계부터 적용합니다. 데이터 품질 이슈는 온톨로지 구축과 병렬로 해결해야 합니다.

## 자기 진단 체크리스트

현재 프로젝트의 건강 상태를 점검하세요.

- [ ] 역량 질문 목록이 명확히 정의되어 있는가?
- [ ] 최근 한 달 내에 실제 사용자가 온톨로지를 활용한 사례가 있는가?
- [ ] 온톨로지를 이해하는 사람이 두 명 이상인가?
- [ ] 데이터 갱신 파이프라인이 자동화되어 있는가?
- [ ] 다음 버전의 범위가 명확히 제한되어 있는가?

모든 항목에 예스라면 건강한 프로젝트입니다. 하나라도 아니오라면 해당 실패 패턴의 위험에 노출되어 있습니다.

## 정리

온톨로지 프로젝트의 실패는 기술적 한계보다 관리적·조직적 함정에서 비롯되는 경우가 많습니다. Ontology Theatre, 범위 크리프, 완벽주의 함정, 도구 우선 접근, 버스 팩터 1, 데이터 품질 과소평가는 사전에 인식하면 예방할 수 있습니다. 마지막 챕터에서는 2025~2027년 로드맵과 의사결정 프레임워크로 이 코스를 마무리합니다.
