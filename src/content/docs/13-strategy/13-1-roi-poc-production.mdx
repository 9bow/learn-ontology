## 왜 ROI가 어려운가

지식 그래프와 온톨로지 프로젝트의 비즈니스 가치는 실재하지만, 측정하기 어렵습니다. 그 이유는 다음과 같습니다.

**간접 가치:** 지식 그래프는 직접 수익을 만들기보다는 다른 시스템의 성능을 향상시킵니다. 검색 정확도 향상, 데이터 통합 시간 단축, 규제 보고 오류 감소 같은 간접 가치를 화폐 단위로 환산하기 어렵습니다.

**장기 투자:** 온톨로지 설계, 데이터 매핑, 품질 관리는 초기 투자가 크고, 가치는 시간이 지남에 따라 복리로 증가합니다. 단기 ROI 기준을 적용하면 투자 결정이 어렵습니다.

**기여도 분리:** 지식 그래프가 기여한 성과를 다른 시스템의 개선과 분리해 측정하기가 쉽지 않습니다.

## ROI 측정 프레임워크

그럼에도 ROI를 측정해야 합니다. 다음 프레임워크가 실무에서 사용됩니다.

**비용 절감 측정:**

| 비용 항목 | 측정 방법 |
|---------|---------|
| 데이터 통합 공수 | 온톨로지 도입 전후 데이터 통합 작업 시간 비교 |
| 데이터 오류 수정 비용 | SHACL 검증으로 사전 탐지된 오류 수 × 수정 비용 |
| 규제 보고 준비 시간 | 보고서 작성 소요 시간 감소 측정 |
| 중복 데이터 관리 비용 | 중복 제거로 절감된 스토리지·처리 비용 |

**수익 기여 측정:**

| 수익 항목 | 측정 방법 |
|---------|---------|
| 검색 정확도 향상 | A/B 테스트: 지식 그래프 적용 전후 검색 클릭률 |
| 추천 시스템 개선 | 추천 전환율 변화 |
| 신규 인사이트 발견 | 그래프에서 발견한 패턴으로 창출한 비즈니스 기회 |

## POC 설계 원칙

성공적인 POC(Proof of Concept)는 프로덕션 이행의 기초가 됩니다. 실패하는 POC의 공통 패턴과 성공 원칙을 정리합니다.

**실패 패턴:**
- 너무 광범위한 범위 설정 ("모든 데이터를 통합하자")
- 명확한 성공 기준 없음 ("뭔가 유용한 것이 나오겠지")
- 도메인 전문가 없이 기술팀만 참여
- POC 데이터가 실제 프로덕션 데이터와 너무 다름

**성공 원칙:**
- **단일 비즈니스 문제에 집중:** "공급업체 리스크 평가 시간을 30% 단축한다"처럼 구체적인 목표를 설정합니다.
- **실제 데이터 사용:** 익명화한 실제 데이터로 POC를 진행합니다. 가상 데이터는 실제 문제를 숨깁니다.
- **측정 가능한 KPI:** 시작 전에 성공/실패를 판단할 지표를 정의합니다.
- **도메인 전문가 참여:** 온톨로지 설계에서 도메인 전문가의 검토가 필수입니다.

## Crawl-Walk-Run 단계 모델

프로덕션 이행은 단계적으로 접근하는 것이 안전합니다.

### Crawl 단계 (1~3개월)
**목표:** 핵심 도메인의 온톨로지 초안 작성, 단일 데이터 소스 연결, 기본 질의 동작 확인

**체크리스트:**
- [ ] 핵심 클래스 20개 이하로 온톨로지 초안 완성
- [ ] 1~2개 데이터 소스의 R2RML 매핑 완성
- [ ] SPARQL 기본 질의 10개 검증
- [ ] 도메인 전문가 피드백 반영

**피해야 할 것:** 이 단계에서 완전한 온톨로지를 만들려는 욕심. 불완전하지만 동작하는 것이 완전하지만 동작 안 하는 것보다 낫습니다.

### Walk 단계 (3~9개월)
**목표:** 다중 데이터 소스 통합, 데이터 품질 파이프라인 구축, 첫 실사용자 확보

**체크리스트:**
- [ ] 3개 이상 데이터 소스 통합
- [ ] SHACL 검증 자동화
- [ ] 데이터 갱신 파이프라인 운영 (일별 또는 주별)
- [ ] 내부 사용자 5명 이상이 정기적으로 사용

**핵심 과제:** 온톨로지 거버넌스 프로세스 수립. 누가 클래스를 추가하고, 변경을 어떻게 승인하는가.

### Run 단계 (9개월~)
**목표:** 프로덕션 SLA 충족, 여러 팀의 의존, 지속적 확장

**체크리스트:**
- [ ] 99% 이상 가용성 보장
- [ ] 쿼리 응답 시간 SLA 충족 (예: p95 < 2초)
- [ ] 온톨로지 변경 관리 프로세스 운영
- [ ] 데이터 계보(lineage) 추적 가능

## 팀 구성

프로덕션 수준의 지식 그래프 운영에 필요한 역할입니다.

| 역할 | 주요 책임 | 필요 역량 |
|------|---------|---------|
| 온톨로지 엔지니어 | 온톨로지 설계·유지보수 | OWL, SHACL, 도메인 지식 |
| 데이터 엔지니어 | ETL 파이프라인, 매핑 | R2RML, SPARQL, Python |
| 도메인 전문가 | 개념 정의 검토, 요구사항 | 해당 업무 도메인 |
| 플랫폼 엔지니어 | 트리플 스토어 운영, 성능 | DevOps, 분산 시스템 |
| 제품 관리자 | 우선순위, 사용자 요구사항 | 프로젝트 관리 |

소규모 팀에서는 한 사람이 여러 역할을 담당하지만, 온톨로지 엔지니어와 도메인 전문가 역할은 분리하는 것이 좋습니다. 한 사람이 모두 담당하면 설계 검증의 독립성이 사라집니다.

## 기술 부채 관리

POC를 빠르게 만들다 보면 기술 부채가 쌓입니다.

- **온톨로지 부채:** 미결 정의, 중복 클래스, 불명확한 범위
- **데이터 부채:** 미검증 매핑, 출처 불명 트리플
- **인프라 부채:** 수동 배포, 테스트 없는 변경

Run 단계로 이행하기 전에 이 부채를 명시적으로 식별하고 상환 계획을 세워야 합니다.

---

## 정리

지식 그래프 프로젝트의 ROI는 측정하기 어렵지만 측정하지 않으면 경영진 지지를 유지하기 어렵습니다. POC에서 프로덕션으로의 이행은 단계적으로 접근하고, 각 단계에서 명확한 성공 기준을 가져야 합니다. 다음 챕터에서는 데이터 거버넌스와 보안 설계를 살펴봅니다.
