## 2025년 현재의 위치

이 코스를 작성한 시점(2025년 기준)에서 지식 그래프 생태계는 세 가지 힘이 동시에 작용하는 상태입니다.

**LLM의 부상:** ChatGPT 이후 LLM이 주목받으면서 지식 그래프의 역할에 대한 논쟁이 일었습니다. "LLM이 모든 것을 기억하면 지식 그래프가 필요 없지 않나?"라는 질문입니다. 결론은 아직 열려 있지만, 현재까지의 경험은 LLM의 환각(hallucination), 최신성, 정밀한 다단계 추론에서 지식 그래프가 보완적 역할을 한다는 방향을 가리킵니다.

**규제 압력:** EU AI Act, GDPR, EU 택소노미, CSDDD 등 데이터의 출처·의미·거버넌스를 요구하는 규제가 강화되고 있습니다. 온톨로지는 이 요건을 충족하는 기술적 기반을 제공합니다.

**표준 성숙:** RDF-star, SPARQL-star, GQL(ISO 39075) 등 차세대 표준이 정착하면서 도구 생태계가 안정화되고 있습니다.

## 2025~2027년 주요 트렌드

### 트렌드 1: LLM-KG 통합의 성숙

2024~2025년에 GraphRAG(Microsoft), KG-enhanced LLM 논문들이 쏟아졌습니다. 2025~2027년에는 이 패턴들이 프로덕션에서 검증되고 표준화될 것으로 예상됩니다.

예상 발전 방향:
- **Text-to-SPARQL 자동화:** 자연어 질문을 SPARQL로 변환하는 모델의 정확도가 충분한 수준에 도달해 실무 적용이 가능해집니다
- **온톨로지 자동 생성:** 텍스트 문서에서 온톨로지 초안을 자동 생성하는 LLM 파이프라인이 실용화됩니다. 사람의 검토와 수정은 여전히 필요하지만, 초안 생성 비용이 크게 낮아집니다
- **Neuro-symbolic 추론:** LLM의 유연한 언어 이해와 지식 그래프의 논리적 추론을 결합하는 아키텍처가 다양하게 시도됩니다

### 트렌드 2: 도메인별 표준 온톨로지 확산

FIBO(금융), SNOMED CT(의료), IEC CDD(제조) 같은 도메인 표준이 산업 전반으로 확산됩니다. 2025~2027년에는 다음 분야에서 표준화가 가속될 것으로 예상됩니다.

- **ESG/지속가능성:** EU 택소노미, ESRS를 기반으로 한 공통 ESG 온톨로지
- **법률/컴플라이언스:** 계약 분석, 규제 추적을 위한 법률 온톨로지
- **공급망:** CSDDD 대응을 위한 공급망 실사 온톨로지

### 트렌드 3: 엣지 컴퓨팅과 경량 온톨로지

IoT와 디지털 트윈의 확산으로 온톨로지가 클라우드가 아닌 엣지 기기에서 실행되어야 하는 수요가 증가합니다. OWL DL 전체를 엣지에서 실행하기는 어렵지만, RDFS 수준의 경량 추론이나 SHACL 검증은 현실적입니다.

### 트렌드 4: 지식 그래프 자동화 도구

온톨로지 매핑, 엔티티 해소, 데이터 품질 검증을 자동화하는 도구가 성숙합니다. 이는 지식 그래프 구축의 진입 장벽을 낮추지만, 동시에 자동화 결과를 검증하는 도메인 전문 지식의 중요성이 더 높아집니다.

## 의사결정 프레임워크

조직이 온톨로지·지식 그래프 도입 여부와 방법을 결정하는 프레임워크입니다.

### 1단계: 필요성 진단

다음 중 하나라도 해당하면 도입을 검토할 필요가 있습니다.

- [ ] 여러 시스템의 데이터를 통합하는 데 반복적으로 많은 시간이 걸린다
- [ ] 같은 개념을 부서마다 다르게 정의해 혼란이 생긴다
- [ ] 규제 보고에서 데이터 출처를 추적하기 어렵다
- [ ] 다단계 관계 조회(예: 공급망 3단계, 소유 구조 계층)가 필요하다
- [ ] AML, ESG, 리스크 집계처럼 복잡한 관계 패턴 탐지가 필요하다

### 2단계: 적합한 도입 수준 선택

| 수준 | 설명 | 언제 적합한가 |
|------|------|------------|
| SKOS 어휘 | 공통 용어 사전 구축 | 부서 간 용어 통일이 주 목적 |
| SHACL 스키마 | 데이터 품질 검증 | 기존 RDF 데이터의 품질 관리 |
| RDFS 온톨로지 | 계층 구조와 기본 추론 | 단순한 분류 체계와 상속 필요 |
| OWL 2 온톨로지 | 완전한 추론 가능 온톨로지 | 복잡한 도메인 제약과 자동 추론 필요 |
| 지식 그래프 플랫폼 | 대규모 통합 + 쿼리 | 엔터프라이즈급 데이터 통합 |

### 3단계: 기술 스택 선택

요구사항에 따른 기술 스택 선택 가이드입니다.

```
추론이 중요한가?
├── Yes → OWL 2 + RDFox, HermiT, Stardog
└── No  → SHACL + GraphDB, Amazon Neptune, Jena

데이터 규모는?
├── 수억 트리플 이상 → GraphDB Enterprise, Neptune, Stardog
└── 수천만 이하 → Jena TDB, GraphDB Free, Oxigraph

LLM 통합이 필요한가?
├── Yes → LangChain KG 모듈, LlamaIndex KG 인덱스 검토
└── No  → 표준 SPARQL 엔드포인트

클라우드 우선인가?
├── AWS → Amazon Neptune
├── Azure → Azure Cosmos DB (Gremlin API)
└── 멀티클라우드/온프레미스 → StarDog, GraphDB
```

### 4단계: 리스크 평가

| 리스크 | 낮음 | 높음 |
|-------|------|------|
| 도메인 복잡도 | 단일 도메인, 명확한 경계 | 다수 도메인, 복잡한 관계 |
| 데이터 품질 | 정제된 소스, 표준 형식 | 레거시 시스템, 다양한 형식 |
| 팀 역량 | OWL·SPARQL 경험자 있음 | 팀 내 전문가 없음 |
| 경영진 지원 | 명확한 스폰서십 | 불명확한 지원 |

리스크가 높을수록 작은 POC에서 시작하고, 외부 전문가 지원을 고려합니다.

## 이 코스에서 배운 것들의 연결

이 코스의 13개 섹션이 어떻게 연결되는지 종합합니다.

**기초(01~05):** 온톨로지의 철학적 기원부터 OWL, SPARQL, 개발 방법론까지. 이 기초 없이 도구와 사례를 이해하기 어렵습니다.

**플랫폼(06~07):** Palantir Foundry·AIP는 온톨로지를 엔터프라이즈 AI의 안전한 기반으로 사용하는 대표 사례입니다. 기술 스택 선택은 요구사항에서 시작해야 합니다.

**AI 결합(08):** GraphRAG, Text-to-SPARQL, Neuro-symbolic AI는 지식 그래프가 LLM 시대에도 여전히 가치 있는 이유입니다.

**산업 사례(09~12):** 의료(SNOMED CT, FHIR), 금융(FIBO, AML, ESG), 제조(디지털 트윈, 공급망), 한국(ETRI, 네이버·카카오)의 사례는 추상적 개념이 실제로 어떻게 적용되는지를 보여줍니다.

**전략(13):** ROI 측정, 거버넌스, 조직 변화 관리, 실패 패턴 회피, 로드맵. 기술을 아는 것과 조직에서 성공적으로 구현하는 것은 다른 역량입니다.

## 시작하는 법

이 코스를 마친 후 첫 번째 실천 단계를 제안합니다.

**이번 주:** 자신의 조직에서 가장 반복적으로 발생하는 데이터 통합 문제를 하나 찾습니다. 역량 질문 형태로 적어봅니다.

**이번 달:** Protégé를 설치하고, 찾은 문제 도메인의 클래스 10개, 속성 5개로 미니 온톨로지를 만들어봅니다.

**3개월 내:** 소규모 팀원 한두 명과 함께 실제 데이터 샘플을 온톨로지에 매핑하고, 역량 질문 하나에 SPARQL로 답하는 POC를 완성합니다.

온톨로지는 선언이 아니라 실천입니다. 작고 구체적인 문제에서 시작하세요.

## 정리

2025~2027년 지식 그래프 기술은 LLM 통합, 도메인 표준 확산, 자동화 도구 성숙의 방향으로 발전할 것입니다. 도입 의사결정은 필요성 진단 → 수준 선택 → 기술 스택 → 리스크 평가의 순서로 접근합니다. 이 코스에서 배운 개념들은 독립적이 아니라 서로 연결된 하나의 생태계입니다. 전략 섹션의 마지막인 이 챕터에서 배운 내용을 퀴즈로 확인하세요.
