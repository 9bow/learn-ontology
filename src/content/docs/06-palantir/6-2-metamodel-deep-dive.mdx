## 메타모델이란

메타모델은 "모델의 모델"입니다. Foundry Ontology의 메타모델은 Object Type, Link Type, Action Type 자체를 어떻게 정의하고 조합하는가에 관한 규칙 체계입니다.

이 챕터에서는 앞 챕터의 기초 개념을 넘어, 실제 엔터프라이즈 배포에서 사용하는 고급 패턴을 분석합니다.

## Object Type 메타모델 심층 분석

### 속성 타입 시스템

Foundry Object Type의 속성은 다음 타입을 지원합니다.

| 타입 | 예시 | 비고 |
|---|---|---|
| `string` | 항공기 번호, 이름 | 최대 길이 설정 가능 |
| `integer` / `long` | 수량, ID | 범위 제약 가능 |
| `double` | 무게, 거리 | 부동소수점 |
| `boolean` | 활성 여부 | true/false |
| `date` | 제조일 | ISO 8601 |
| `timestamp` | 최종 수정일 | UTC 기준 |
| `geoshape` | 위치, 경로 | GeoJSON 호환 |
| `attachment` | 문서, 이미지 | 파일 참조 |
| `array<T>` | 태그 목록 | 기본 타입의 배열 |
| `struct` | 중첩 속성 | 복합 값 객체 |

### 파생 속성(Derived Properties)

파생 속성은 다른 속성이나 Link를 통해 계산되는 읽기 전용 속성입니다.

```
Object Type: Aircraft
├── properties (직접 저장):
│   ├── total_flight_hours: double
│   └── last_maintenance_date: date
│
└── derived properties (계산):
    ├── days_since_maintenance:
    │   계산식: today() - last_maintenance_date
    └── maintenance_overdue:
        계산식: days_since_maintenance > 365
```

파생 속성은 데이터를 중복 저장하지 않고 비즈니스 로직을 온톨로지 레이어에 위치시키는 방법입니다.

### Object Type 시리즈(Time Series)

Foundry는 시계열 데이터를 위한 특수 Object Type 패턴을 지원합니다.

```
Object Type: AircraftSensor
├── primaryKey: sensor_id
├── properties:
│   ├── sensor_type: string
│   └── unit: string
└── timeSeries:
    └── readings: double (timestamp 인덱스)
```

시계열 속성은 `aircraft.sensor.readings.last(24h).avg()` 같은 시간 범위 집계 쿼리를 지원합니다.

## Link Type 메타모델 심층 분석

### 카디널리티 패턴

| 카디널리티 | 예시 | 구현 방식 |
|---|---|---|
| 1:1 | Employee ↔ Badge | 단일 외래 키 |
| 1:N | Aircraft → MaintenanceTasks | 외래 키 배열 또는 참조 테이블 |
| N:M | Task ↔ Parts | 브리지 테이블 |
| N:1 | Flights → Airport (출발지) | 외래 키 |

### 복합 Link Type (Bridge Object)

N:M 관계에서 관계 자체에 속성이 필요한 경우, 브리지 Object Type을 사용합니다.

```
직접 연결이 불가능한 패턴:
Aircraft --[requires_part]--> Part

이유: "어떤 작업에서 몇 개의 부품이 필요한가?"를 표현할 수 없음

해법: 브리지 Object Type 도입

Aircraft --[hasMaintenanceTask]--> MaintenanceTask
MaintenanceTask --[requiresPart]--> PartRequirement
PartRequirement --[specifiesPart]--> Part

PartRequirement properties:
├── quantity: integer
├── priority: enum
└── substitutable: boolean
```

### 다중 홉 탐색 설계

Foundry는 Link를 통한 다중 홉 탐색을 지원합니다. 설계 시 탐색 경로를 명시적으로 고려합니다.

```
탐색 시나리오: "이 항공사의 모든 정비 작업 중 지연된 것은?"

탐색 경로:
Airline
  └─[operatesAircraft]─▶ Aircraft
      └─[hasMaintenanceTask]─▶ MaintenanceTask
          ↓ filter: status == "Delayed"
```

3홉 이상의 탐색은 성능 저하를 유발할 수 있습니다. 자주 사용하는 다중 홉 경로는 파생 Link Type이나 Object View로 단축합니다.

### Object View (가상 Object Type)

Object View는 기존 Object Type과 Link의 조합으로 새로운 "가상" Object Type을 정의합니다. 실제 데이터를 중복 저장하지 않고 편의성을 제공합니다.

```
Object View: DelayedMaintenanceTask
← MaintenanceTask where status == "Delayed"

추가 파생 속성:
├── days_delayed: today() - scheduled_date
└── affected_aircraft: link(Aircraft)
```

## Action Type 메타모델 심층 분석

### 검증 로직 계층

Action Type의 검증은 세 계층으로 구성됩니다.

**레이어 1: 파라미터 타입 검증**
입력값의 타입이 선언과 일치하는지 자동으로 확인합니다. 날짜 필드에 문자열이 들어오면 즉시 거부됩니다.

**레이어 2: 비즈니스 규칙 검증**
사용자가 정의한 비즈니스 로직을 실행합니다.

```
Action Type: ApproveMaintenancePlan
검증 규칙:
├── 규칙 1: task.scheduled_date > today()
│   실패 메시지: "과거 날짜로 정비를 승인할 수 없습니다"
├── 규칙 2: task.assigned_technician != null
│   실패 메시지: "담당 기술자가 지정되어야 합니다"
└── 규칙 3: currentUser.hasRole("MaintenanceManager")
    실패 메시지: "정비 관리자만 승인할 수 있습니다"
```

**레이어 3: 외부 검증 (웹훅)**
외부 시스템에 검증 요청을 보내고 응답을 기다립니다.

```
Action Type: OrderPart
웹훅 검증:
├── URL: https://erp.company.com/api/validate-order
├── 요청: { part_id, quantity, urgency }
└── 응답: { approved: boolean, reason: string }
```

외부 ERP, 규정 준수 시스템과 실시간 연동이 가능합니다.

### Action Effect 유형

| Effect 유형 | 설명 | 예시 |
|---|---|---|
| `modifyObject` | Object 속성 변경 | status 업데이트 |
| `createObject` | 새 Object 인스턴스 생성 | 작업 지시서 생성 |
| `deleteObject` | Object 논리 삭제 | 레코드 비활성화 |
| `addLink` | Link 연결 추가 | 항공기에 정비 작업 할당 |
| `removeLink` | Link 연결 제거 | 할당 해제 |
| `createAndLink` | 생성 + 연결 동시 | 새 작업 + 항공기 연결 |
| `callFunction` | Foundry Function 호출 | 복잡한 연산 |
| `webhookEffect` | 외부 시스템 호출 | ERP 업데이트 |

### 감사 로그(Audit Trail)

모든 Action 실행은 자동으로 감사 로그에 기록됩니다.

```
감사 로그 레코드:
├── actionType: "RescheduleMaintenanceTask"
├── executedBy: "user@company.com"
├── executedAt: "2024-03-15T09:23:41Z"
├── parameters: { task: "TASK-001", newDate: "2024-04-01" }
├── status: "SUCCESS"
└── effects: [
    { type: "modifyObject", objectType: "MaintenanceTask",
      objectId: "TASK-001", field: "scheduled_date",
      oldValue: "2024-03-20", newValue: "2024-04-01" }
  ]
```

규제 산업(항공, 의료, 금융)에서 변경 이력 추적이 필수인 경우, Action Type 기반 아키텍처가 이를 자동으로 해결합니다.

## Foundry 메타모델 vs OWL 메타모델 비교

| 개념 | OWL | Foundry Ontology |
|---|---|---|
| 클래스 | `owl:Class` | Object Type |
| 인스턴스 | 개별 IRI | Object (데이터셋 행) |
| 객체 속성 | `owl:ObjectProperty` | Link Type |
| 데이터 속성 | `owl:DatatypeProperty` | Object Type Property |
| 공리 | 논리 공리 (DL) | 검증 규칙 (Action) |
| 추론 | DL 추론기 | 파생 속성, Functions |
| 변경 | SPARQL Update | Action Type |
| 접근제어 | 외부 시스템 | 내장 (Object/Action 레벨) |
| 데이터소스 | 분리 | Object Type에 내장 |

핵심 차이는 **운영성(operationality)** 입니다. OWL은 지식 표현과 추론에 최적화되어 있고, Foundry Ontology는 데이터 통합과 운영 워크플로에 최적화되어 있습니다.

## 메타모델 설계 안티패턴

**안티패턴 1: 너무 많은 Object Types**

테이블마다 Object Type을 만들면 온톨로지가 데이터 카탈로그로 전락합니다. 비즈니스적으로 독립적인 엔티티만 Object Type으로 정의합니다.

**안티패턴 2: 검증 없는 Action Types**

검증 규칙 없이 Action을 정의하면 데이터 무결성이 깨집니다. 모든 Action에는 최소한 권한 검사와 필수 파라미터 검증을 포함합니다.

**안티패턴 3: 순환 Link**

A→B→A 형태의 순환 Link는 다중 홉 탐색 시 무한 루프를 유발할 수 있습니다. 순환이 필요하다면 명시적 깊이 제한을 설정합니다.

**안티패턴 4: 속성 과다**

한 Object Type에 수백 개의 속성을 정의하면 관리가 불가능해집니다. 자주 사용하지 않는 속성은 별도 Object Type으로 분리하고 Link로 연결합니다.

## 정리

Foundry Ontology 메타모델은 기업 운영 환경에 최적화된 설계입니다.

- **Object Type** : 파생 속성, 시계열, Object View로 표현력 확장
- **Link Type** : 브리지 Object로 N:M 관계 속성 표현, 다중 홉 경로 설계
- **Action Type** : 3계층 검증(타입·비즈니스·외부), 다양한 Effect 유형, 자동 감사 로그

OWL과 비교하면 추론 능력 대신 운영성과 접근제어를 강화한 트레이드오프입니다. 다음 챕터에서는 Functions, Pipelines, Workshop을 통해 이 온톨로지를 어떻게 활용하는지 살펴봅니다.
