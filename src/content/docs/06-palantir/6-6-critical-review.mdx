## 비판적 검토의 필요성

앞선 챕터들이 Foundry Ontology의 기능과 사례를 설명했다면, 이 챕터는 반대 방향에서 묻습니다. "이 기술을 도입해야 하는가, 도입한다면 무엇을 감수해야 하는가?"

좋은 기술 결정은 마케팅 자료가 아니라 트레이드오프 분석에서 나옵니다. Palantir는 강력한 마케팅 내러티브를 가진 회사입니다. 이를 균형 있게 검토하는 것이 이 챕터의 목적입니다.

---

## 벤더 락인 분석

### 락인의 세 계층

Foundry를 깊이 도입하면 세 계층의 의존성이 형성됩니다.

**계층 1: 데이터 모델 락인**

Foundry Ontology의 Object Type, Link Type, Action Type은 Palantir 독점 형식입니다. W3C OWL, RDF, SPARQL이 아닙니다. 이 모델로 수년간 구축한 온톨로지를 다른 플랫폼으로 이전하려면 전체 재설계가 필요합니다.

비교: OWL 기반 온톨로지는 Protégé, GraphDB, Stardog, Jena 등 다양한 오픈소스·상용 도구로 이전 가능합니다.

**계층 2: 로직 락인**

Foundry Functions는 Foundry SDK에 의존하는 TypeScript입니다. Foundry 런타임 없이는 실행되지 않습니다. AIP Logic의 에이전트 정의도 Palantir 독점 포맷입니다.

**계층 3: 운영 프로세스 락인**

Workshop 대시보드, Pipeline, Action 기반으로 업무 프로세스가 재구성되면, 플랫폼 교체 시 단순한 소프트웨어 전환을 넘어 **업무 방식 자체의 재설계** 가 필요합니다.

### 이탈 비용 추정

이탈 비용은 다음 요소의 합입니다.

| 요소 | 설명 |
|---|---|
| 데이터 추출 | Foundry 데이터셋을 표준 포맷으로 내보내기 |
| 온톨로지 재설계 | Foundry 모델을 OWL/LPG로 변환 |
| 로직 재구현 | Functions를 다른 플랫폼 언어로 재작성 |
| UI 재구축 | Workshop 대시보드 대체 |
| 사용자 재교육 | 새로운 도구 학습 |
| 운영 중단 리스크 | 전환 기간 동안의 불연속성 |

Palantir는 데이터 추출 API를 제공하므로 데이터 자체는 비교적 이전하기 쉽습니다. 그러나 데이터 위에 구축된 **의미론적 모델과 로직** 은 이전이 어렵습니다.

### 표준 기반 대안과 비교

| 항목 | Foundry Ontology | OWL + SPARQL + 오픈소스 |
|---|---|---|
| 데이터 모델 | 독점 | W3C 표준 |
| 질의 언어 | 독점 API | SPARQL (표준) |
| 추론 | Functions (독점) | 오픈소스 추론기 (HermiT, RDFox) |
| 이전 가능성 | 어려움 | 용이 |
| 통합 용이성 | Foundry 생태계 내 우수 | 다양한 도구와 통합 가능 |
| 운영 복잡성 | 낮음 (플랫폼이 처리) | 높음 (직접 구성 필요) |

## 비용 구조

### 공개 정보의 한계

Palantir는 공개 가격표를 제공하지 않습니다. 계약은 기업별 협상으로 이루어지며, 세부 조건은 기밀입니다. Palantir 10-K에서 확인할 수 있는 것은 전체 매출, 고객 수, 평균 계약 규모 정도입니다.

**10-K에서 확인된 사실 (2023년 기준):**
- 상업 부문 고객당 평균 매출: 약 $3.3M/년 (상업 고객 총 수와 매출에서 산출)
- 정부 부문 계약은 별도 (일반적으로 더 대규모)

이 수치는 전체 평균이며, 개별 계약은 수십만 달러에서 수억 달러까지 분포합니다.

### 비용 구조의 특성

Foundry 비용은 일반적으로 다음을 포함합니다.

- **플랫폼 라이선스**: 기본 접근권
- **사용자 시트**: 활성 사용자 수 기반
- **데이터 볼륨**: 처리·저장 데이터량
- **컴퓨팅**: Pipeline, Function 실행 비용
- **구현 서비스**: 초기 도입 컨설팅 (Forward Deployed Engineer)

Forward Deployed Engineer(FDE) 모델은 Palantir의 특징적 판매 방식입니다. Palantir 엔지니어가 고객 현장에 상주하며 구현을 지원합니다. 이는 빠른 가치 실현을 돕지만, FDE 비용이 별도로 발생하고 Palantir에 대한 기술 의존도를 높이는 효과가 있습니다.

## "OS for Enterprise" 주장 검증

Palantir는 Foundry Ontology를 "Enterprise의 운영체제"라고 마케팅합니다. 이 주장을 기술적으로 분석합니다.

### 주장의 근거

Palantir가 이 비유를 사용하는 근거:

1. **추상화 레이어**: OS가 하드웨어를 추상화하듯, Foundry는 데이터 인프라를 추상화합니다
2. **공유 리소스 관리**: OS가 메모리·CPU를 관리하듯, Foundry는 데이터·권한을 관리합니다
3. **애플리케이션 플랫폼**: OS 위에 앱이 실행되듯, Foundry 위에 Workshop 앱이 실행됩니다

### 주장의 한계

**한계 1: 이식성 부재**

실제 OS는 다양한 하드웨어 위에서 실행됩니다. Foundry는 Palantir 클라우드 또는 고객 인프라에서 실행되지만, "다른 Foundry"는 없습니다. 경쟁사 플랫폼으로 "앱"을 이식할 수 없습니다.

**한계 2: 표준 API 부재**

실제 OS는 POSIX 같은 표준 API를 제공합니다. Foundry API는 독점이며 외부에서 표준화되지 않았습니다. 서드파티 생태계가 OS 수준으로 형성되어 있지 않습니다.

**한계 3: 범위의 제한**

OS는 모든 소프트웨어의 기반입니다. Foundry는 데이터·분석·운영 워크플로 영역에 특화됩니다. ERP, CRM, 제조 실행 시스템(MES) 등 기존 엔터프라이즈 소프트웨어를 대체하지 않고 그 위에 레이어로 추가됩니다.

### 더 정확한 비유

"Enterprise의 의미론적 데이터 통합 및 운영 레이어"가 더 정확합니다. 강력한 플랫폼이지만, OS 수준의 보편성과 이식성은 없습니다.

## NHS 논란과 공공 데이터 거버넌스

NHS FDP는 Palantir 도입의 논쟁적 측면을 잘 보여줍니다.

### 주요 우려사항 (공개 토론 기반)

**프라이버시 우려**: openDemocracy, 의료 전문가 단체, 시민 단체는 NHS 환자 데이터의 상업 플랫폼 처리에 우려를 표명했습니다. "환자 데이터가 Palantir의 AI 학습에 사용될 수 있는가?" 같은 질문이 제기되었습니다.

**투명성 문제**: 계약 조건이 전면 공개되지 않아 의회와 시민 단체의 자료 공개 요청이 있었습니다.

**벤더 락인 우려**: 장기 계약으로 NHS가 Palantir에 구조적으로 의존하게 될 수 있다는 지적입니다.

### NHS의 공식 입장

NHS England는 공식 웹사이트에 다음을 명시합니다.

- 환자 데이터는 NHS 통제 하에 있음
- Palantir는 데이터를 상업적으로 활용할 수 없음
- 독립적 감사 체계 운영

이 논란은 Palantir 기술의 문제라기보다 공공 데이터 거버넌스에 관한 더 넓은 사회적 논쟁입니다. 그러나 Palantir의 과거 정부 계약 이력(팔란티어라는 이름 자체의 감시 연상)이 이 논란을 증폭시킵니다.

## 도입 결정 프레임워크

Foundry 도입을 평가할 때 고려해야 할 질문입니다.

### 도입에 유리한 조건

- 다수의 이기종 데이터소스 통합이 핵심 과제인가?
- 비개발자(운영 담당자)의 데이터 접근이 중요한가?
- 빠른 가치 실현이 장기 이식성보다 중요한가?
- 예산이 엔터프라이즈 수준으로 충분한가?
- 표준 규정 준수보다 운영 통합이 우선인가?

### 도입에 불리한 조건

- W3C 표준 준수가 의무인 공공·학술 영역인가?
- 장기적으로 벤더 독립성이 전략적으로 중요한가?
- 오픈소스 스택으로 충분히 해결 가능한 문제인가?
- 계약 비용이 조직 규모에 비해 과도한가?
- 데이터 거버넌스 규정이 독점 플랫폼 사용을 제한하는가?

### 대안 평가

| 요구사항 | Palantir Foundry | 오픈소스 대안 |
|---|---|---|
| 데이터 통합 | Foundry Pipeline | Apache Kafka + Spark |
| 시맨틱 레이어 | Foundry Ontology | OWL + GraphDB/Stardog |
| 비즈니스 UI | Workshop | Apache Superset / Metabase |
| LLM 에이전트 | AIP Logic | LangChain + 자체 구현 |
| 통합 관리 | 단일 플랫폼 | 별도 조합 (복잡도 증가) |

오픈소스 조합은 낮은 라이선스 비용과 높은 이식성을 제공하지만, 통합·운영의 복잡도가 현저히 높습니다.

## 정리

Palantir Foundry Ontology는 기술적으로 강력한 플랫폼입니다. 그러나 균형 잡힌 평가를 위해 다음 사항을 명확히 인식해야 합니다.

**강점:**
- 이기종 데이터의 신속한 의미론적 통합
- 비개발자 친화적 운영 레이어
- Ontology 기반 LLM 에이전트의 구조적 안전성

**약점·위험:**
- 독점 모델로 인한 높은 이탈 비용
- 불투명한 가격 구조
- "OS for Enterprise" 주장의 기술적 과장
- 공공 데이터 거버넌스에서의 신뢰 문제

기술 결정은 기능 목록이 아니라 **트레이드오프** 분석에 근거해야 합니다. Foundry가 최선인 조직이 있고, 표준 기반 오픈소스 스택이 더 적합한 조직이 있습니다.
