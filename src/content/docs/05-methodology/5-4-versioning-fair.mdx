## 온톨로지는 왜 변하는가

온톨로지는 완성되는 순간이 없습니다. 도메인 지식이 발전하고, 새로운 사용 사례가 추가되며, 오류가 발견됩니다. 소프트웨어 시스템과 마찬가지로 온톨로지도 지속적으로 진화합니다.

문제는 온톨로지 변경이 **하위 시스템 전체에 파급 효과** 를 미친다는 것입니다. 클래스 이름을 바꾸거나 속성을 삭제하면, 해당 온톨로지를 사용하는 SPARQL 쿼리, 추론 규칙, 연동 시스템이 모두 영향을 받습니다.

체계적인 버전 관리와 변경 관리가 필요한 이유입니다.

## OWL 버전 어노테이션

OWL 2는 온톨로지 메타데이터를 위한 표준 어노테이션 속성을 제공합니다.

```turtle
@prefix owl:  <http://www.w3.org/2002/07/owl#> .
@prefix xsd:  <http://www.w3.org/2001/XMLSchema#> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .
@prefix dcterms: <http://purl.org/dc/terms/> .

<https://example.org/ontology/drug>
    a owl:Ontology ;

    # OWL 표준 버전 어노테이션
    owl:versionIRI <https://example.org/ontology/drug/2.1.0> ;
    owl:versionInfo "2.1.0" ;
    owl:priorVersion <https://example.org/ontology/drug/2.0.0> ;
    owl:backwardCompatibleWith <https://example.org/ontology/drug/2.0.0> ;

    # Dublin Core 메타데이터
    dcterms:title "Drug Ontology"@ko ;
    dcterms:description "의약품 도메인 온톨로지"@ko ;
    dcterms:created "2022-01-01"^^xsd:date ;
    dcterms:modified "2024-03-15"^^xsd:date ;
    dcterms:creator <https://orcid.org/0000-0001-2345-6789> ;
    dcterms:license <https://creativecommons.org/licenses/by/4.0/> .
```

### 핵심 OWL 버전 속성

| 속성 | 의미 |
|---|---|
| `owl:versionIRI` | 현재 버전의 고유 IRI |
| `owl:versionInfo` | 사람이 읽는 버전 문자열 |
| `owl:priorVersion` | 이전 버전 IRI |
| `owl:backwardCompatibleWith` | 하위 호환 버전 명시 |
| `owl:incompatibleWith` | 비호환 버전 명시 |

## 시맨틱 버저닝 적용

소프트웨어의 시맨틱 버저닝(SemVer) 규칙을 온톨로지에 적용합니다.

**MAJOR.MINOR.PATCH** 형식:

| 버전 변경 | 의미 | 예시 |
|---|---|---|
| MAJOR 증가 | 하위 호환성 파괴 변경 | 클래스/속성 삭제, URI 변경 |
| MINOR 증가 | 하위 호환 기능 추가 | 새 클래스/속성 추가, 정의 확장 |
| PATCH 증가 | 버그 수정, 주석 개선 | 오타 수정, 레이블 추가 |

```
1.0.0 → 1.0.1  : 레이블 오타 수정 (PATCH)
1.0.1 → 1.1.0  : 새로운 클래스 DrugInteraction 추가 (MINOR)
1.1.0 → 2.0.0  : hasIngredient를 hasActiveIngredient로 URI 변경 (MAJOR)
```

## 변경 유형 분류

### 하위 호환성 유지 변경 (비파괴)

이런 변경은 기존 시스템에 영향을 주지 않습니다.

- 새 클래스 추가
- 기존 클래스의 하위 클래스 추가
- 새 속성 추가
- 레이블·주석·예시 추가
- 새 공리 추가 (단, 추론 결과가 더 많아지는 방향)

### 하위 호환성 파괴 변경 (파괴)

이런 변경은 기존 시스템을 깨뜨릴 수 있습니다.

- 클래스 또는 속성 삭제
- URI 변경 (삭제 후 재추가와 동일)
- 속성의 도메인·범위 변경
- 카디널리티 제약 강화
- 기존 공리 삭제 또는 수정

### 마이그레이션 전략

파괴적 변경을 피할 수 없을 때 사용하는 전략:

```turtle
# 전략 1: deprecated 표시 후 새 URI 병행 유지
ex:hasIngredient
    a owl:ObjectProperty ;
    owl:deprecated true ;
    rdfs:comment "deprecated: ex:hasActiveIngredient를 사용하세요." ;
    owl:equivalentProperty ex:hasActiveIngredient .

# 전략 2: owl:sameAs로 구 URI와 신 URI 연결
ex:OldClass owl:equivalentClass ex:NewClass .
```

## FAIR 원칙과 온톨로지

FAIR 원칙은 2016년 Wilkinson 등이 Scientific Data에 발표한 과학 데이터 관리 지침입니다. 원래 연구 데이터를 대상으로 했지만, 온톨로지와 어휘에 그대로 적용됩니다.

### F — Findable (발견 가능)

온톨로지는 전역적으로 고유한 영속 식별자를 가져야 합니다.

**실천 방안:**

```turtle
# PURLs 또는 w3id.org 영속 URI 사용
<https://w3id.org/myorg/ontology/drug>
    a owl:Ontology ;
    owl:versionIRI <https://w3id.org/myorg/ontology/drug/2.1.0> .
```

- `purl.org` 또는 `w3id.org`를 통한 영속 URI 확보
- LOV(Linked Open Vocabularies), BioPortal 등 레지스트리에 등록
- 검색 엔진이 찾을 수 있도록 콘텐츠 협상(Content Negotiation) 지원

### A — Accessible (접근 가능)

표준 프로토콜로 온톨로지에 접근할 수 있어야 합니다.

**콘텐츠 협상 구현 예시:**

클라이언트가 `Accept: text/turtle`을 보내면 Turtle 형식으로, `Accept: application/rdf+xml`을 보내면 RDF/XML로 응답합니다. Apache/Nginx 설정으로 구현하거나, pyLODE 같은 도구를 사용합니다.

```
# 요청: HTML 브라우저
GET /ontology/drug
Accept: text/html
→ 사람이 읽는 문서 페이지 반환

# 요청: 소프트웨어 에이전트
GET /ontology/drug
Accept: text/turtle
→ Turtle 형식 온톨로지 반환
```

- 온톨로지가 삭제되어도 메타데이터는 접근 가능하게 유지
- HTTP 상태 코드 410 Gone + 포인터로 이동 위치 안내

### I — Interoperable (상호운용 가능)

온톨로지는 표준 형식과 공유 어휘를 사용해야 합니다.

**실천 방안:**

```turtle
# 표준 어휘 재사용
:Drug
    a owl:Class ;
    rdfs:subClassOf <http://semanticscience.org/resource/SIO_010038> ; # ChEBI와 정렬
    owl:equivalentClass <http://purl.obolibrary.org/obo/CHEBI_23367> .
```

- 기존 상위 온톨로지(BFO, RO, PATO)와 정렬
- 도메인 표준 온톨로지와 `owl:equivalentClass`·`owl:equivalentProperty` 매핑
- SKOS `skos:exactMatch`·`skos:closeMatch`로 어휘 간 연결

### R — Reusable (재사용 가능)

온톨로지는 명확한 라이선스와 충분한 메타데이터를 가져야 합니다.

```turtle
<https://w3id.org/myorg/ontology/drug>
    dcterms:license <https://creativecommons.org/licenses/by/4.0/> ;
    dcterms:rights "CC BY 4.0 — 출처 표시 시 자유 사용 가능" ;
    dcterms:creator <https://orcid.org/0000-0001-2345-6789> ;
    dcterms:contributor <https://orcid.org/0000-0002-3456-7890> ;
    dcterms:publisher <https://example.org/organization> ;
    dcterms:source <https://doi.org/10.1234/drug-ontology-paper> ;
    void:vocabulary <http://semanticscience.org/resource/> ;
    dcterms:description """
        의약품 온톨로지. 활성 성분, 투여 경로, 약물 상호작용을
        표현하는 어휘를 제공합니다. KCD-7 분류와 정렬됩니다.
    """@ko .
```

## 온톨로지 변경 문서화

변경 이력을 온톨로지 내부와 외부 문서로 모두 관리합니다.

### 내부: owl:versionInfo와 rdfs:comment

```turtle
ex:hasActiveIngredient
    a owl:ObjectProperty ;
    rdfs:comment """
        v2.0에서 hasIngredient를 대체합니다.
        도메인: Drug, 범위: ChemicalSubstance.
        변경 이유: 비활성 성분과의 구분 명확화.
    """@ko .
```

### 외부: CHANGELOG 문서

```markdown
## [2.1.0] - 2024-03-15
### Added
- DrugInteraction 클래스 추가
- hasSeverity 속성 추가 (domain: DrugInteraction)

### Changed
- Drug의 rdfs:label에 영어 레이블 추가

## [2.0.0] - 2023-09-01
### Breaking Changes
- hasIngredient → hasActiveIngredient (URI 변경)
- OldAdministrationRoute 삭제
```

## 정리

체계적인 버전 관리와 FAIR 준수는 온톨로지를 장기적으로 유지하는 기반입니다.

- `owl:versionIRI`와 시맨틱 버저닝으로 버전을 명확히 관리합니다
- 파괴적 변경은 `owl:deprecated`와 마이그레이션 경로를 함께 제공합니다
- FAIR 원칙: 영속 URI(F), 콘텐츠 협상(A), 표준 어휘 정렬(I), 라이선스·메타데이터(R)
- 변경 이력은 온톨로지 내부와 외부 CHANGELOG에 모두 문서화합니다

다음 챕터에서는 온톨로지 개발에서 가장 어려운 부분 중 하나인 도메인 전문가와의 협업을 다룹니다.
