## 협업이 어려운 이유

온톨로지 개발에서 기술적 도전보다 더 어려운 것은 종종 **사람** 과의 협업입니다. 온톨로지 엔지니어와 도메인 전문가는 서로 다른 언어를 씁니다.

온톨로지 엔지니어는 "클래스의 도메인과 범위가 무엇인가요?"라고 묻습니다. 도메인 전문가는 이 질문 자체를 이해하지 못합니다. 반대로 전문가가 "처방 주기에 따라 달라지는 용량을 표현해야 해요"라고 말하면, 엔지니어는 이것이 얼마나 복잡한 모델링 문제인지 즉시 파악하지 못합니다.

이 간극을 메우는 것이 협업의 핵심 과제입니다.

## 역할 정의

성공적인 온톨로지 프로젝트는 역할을 명확히 분리합니다.

| 역할 | 담당자 | 주요 책임 |
|---|---|---|
| 도메인 전문가 | 의사, 법률가, 엔지니어 등 | 개념·관계·제약의 내용적 정확성 |
| 온톨로지 엔지니어 | 지식 공학자 | 형식 모델링, OWL 구현, 일관성 유지 |
| 프로젝트 관리자 | PM | 일정·우선순위·이해관계자 조율 |
| 최종 사용자 | 시스템 사용자 | CQ 제공, 결과물 수용 테스트 |

## 지식 획득 기법

### 기법 1: 반구조화 인터뷰

전문가의 머릿속 지식을 끌어내는 가장 기본적인 방법입니다. 완전히 열린 질문은 너무 광범위하고, 완전히 닫힌 질문은 유연성이 없습니다. 반구조화 인터뷰는 그 중간입니다.

**효과적인 인터뷰 질문 유형:**

```
개념 탐색:
"[도메인]에서 가장 중요한 사물/개념 10가지를 꼽는다면?"

관계 탐색:
"[개념 A]와 [개념 B]는 어떤 관계인가요?"

제약 탐색:
"[개념 A]가 항상 만족해야 하는 조건이 있나요?"

예외 탐색:
"일반적으로는 그렇지만, 그렇지 않은 경우가 있나요?"

경계 탐색:
"[개념 A]와 [개념 B]의 차이점은 무엇인가요?"
```

**피해야 할 질문:**
- "이 클래스의 속성은 무엇인가요?" (전문가가 이해 못 함)
- 예/아니오 답변만 가능한 폐쇄형 질문

### 기법 2: 카드 소팅(Card Sorting)

중요 용어를 카드(포스트잇)에 하나씩 적고, 전문가에게 비슷한 것끼리 묶으라고 합니다. 그 결과가 자연스러운 분류 체계를 드러냅니다.

**진행 방식:**

1. 인터뷰에서 수집한 용어 30~50개를 카드에 작성
2. 전문가에게 관련 있는 카드를 그룹으로 묶도록 요청
3. 각 그룹에 이름을 붙이도록 요청
4. 그룹 간 관계를 설명하도록 요청

이 과정이 클래스 계층의 초안이 됩니다.

### 기법 3: 프로토타입 기반 검토

온톨로지 엔지니어가 초안을 만들고 전문가에게 검토를 요청합니다. 빈 화면 앞에서 "무엇이 필요한가요?"를 묻는 것보다, 구체적인 모델을 보여주고 "이게 맞나요?"를 묻는 것이 훨씬 효과적입니다.

**중간 표현 활용:** METHONTOLOGY의 표 기반 중간 표현이나 그래픽 다이어그램을 사용합니다. OWL 코드를 직접 보여주는 것은 피합니다.

```
# 전문가에게 보여줄 수 있는 형태
Drug ──[hasActiveIngredient]──▶ ChemicalSubstance
Drug ──[routeOfAdministration]──▶ Route
Drug ──[treatsCondition]──▶ MedicalCondition
```

### 기법 4: 시나리오 워크스루

"실제 업무에서 이 시스템을 어떻게 사용하실 건가요?"라고 묻고, 전문가가 단계별로 시나리오를 설명하도록 합니다.

## 반복적으로 나타나는 실패 패턴

수많은 온톨로지 프로젝트에서 공통적으로 나타나는 실패 패턴이 있습니다. 미리 인식하는 것이 예방의 첫 걸음입니다.

### 실패 패턴 1: 범위 없는 온톨로지 (Ontology Without Scope)

**증상:** "모든 것을 표현하자"는 목표로 시작해 수년간 개발이 지속되지만 실용적 산출물이 없습니다.

**원인:** 초기에 CQ와 범위를 정의하지 않았습니다.

**예방:** 프로젝트 시작 전 CQ 20~30개를 정의하고 범위를 문서화합니다. "v1.0에서 답할 수 있는 질문"과 "향후 버전으로 이연"을 명시적으로 분리합니다.

### 실패 패턴 2: 전문가 없는 온톨로지 (Ontology Without Domain Experts)

**증상:** 온톨로지 엔지니어가 위키백과나 문서를 읽고 혼자 도메인 지식을 정의합니다. 결과물이 실제 업무와 맞지 않습니다.

**원인:** 전문가 참여 비용과 시간을 과소평가했습니다.

**예방:** 전문가 참여를 프로젝트 계획에 명시적으로 포함합니다. 인터뷰는 최소 월 2회, 중간 검토는 마일스톤마다 진행합니다.

### 실패 패턴 3: 위원회 설계 (Design by Committee)

**증상:** 너무 많은 이해관계자가 모든 결정에 관여합니다. 합의가 불가능하고 온톨로지가 아무도 만족하지 못하는 절충안이 됩니다.

**원인:** 의사결정 권한이 불명확합니다.

**예방:** 최종 의사결정자(온톨로지 아키텍트)를 명확히 지정합니다. 전문가는 자문 역할이고, 최종 결정은 아키텍트가 내립니다. 이견은 기록하되 결정은 진행합니다.

### 실패 패턴 4: 구현 종속 설계 (Implementation-Driven Design)

**증상:** 특정 트리플스토어의 한계나 특성에 맞춰 온톨로지를 설계합니다. 시스템을 교체할 때 온톨로지 전체를 재설계해야 합니다.

**원인:** 도구 친숙도가 설계를 지배합니다.

**예방:** 온톨로지 설계는 구현과 독립적으로 진행합니다. 개념화 단계에서는 특정 트리플스토어를 고려하지 않습니다.

### 실패 패턴 5: 완벽주의 함정 (Perfectionism Trap)

**증상:** "완벽한 온톨로지"를 추구하며 배포를 계속 미룹니다. 실제 데이터나 사용자 피드백 없이 설계가 반복됩니다.

**원인:** 온톨로지가 틀리면 모든 것이 무너진다는 과도한 두려움입니다.

**예방:** "완벽한 온톨로지는 없다"는 원칙을 팀이 공유합니다. 불완전해도 v1.0을 배포하고 실제 사용에서 피드백을 수집합니다. 버전 관리(5-4 챕터)가 이 전략을 지원합니다.

## 전문가 피로도 관리

장기 프로젝트에서 도메인 전문가의 참여 의욕이 떨어지는 것은 흔한 문제입니다.

**원인:**
- 인터뷰가 너무 길고 빈번합니다
- 자신의 입력이 결과물에 어떻게 반영되는지 알 수 없습니다
- 기술적 논의가 이해하기 어렵습니다

**대응 전략:**

| 전략 | 실천 방법 |
|---|---|
| 세션 단축 | 인터뷰는 60분 이내, 검토 세션은 30분 이내 |
| 가시적 피드백 | 인터뷰 후 48시간 내 반영 내용 요약 공유 |
| 비기술적 표현 | OWL 대신 다이어그램과 자연어 문서 사용 |
| 가치 공유 | "당신의 지식으로 이런 질문이 가능해졌습니다" 구체적으로 보여주기 |
| 인정 | 온톨로지 크레딧에 전문가 이름 명시 |

## 협업 성숙도 모델

온톨로지 팀의 협업 역량을 평가하는 간단한 모델입니다.

| 수준 | 특징 |
|---|---|
| 레벨 1: 임시적 | 전문가 참여가 비정기적, 문서화 없음 |
| 레벨 2: 반복적 | 정기 인터뷰 계획, 기본 문서화 |
| 레벨 3: 정의된 | CQ 기반 설계, 역할 명확, 검토 프로세스 |
| 레벨 4: 관리된 | 변경 관리, FAIR 준수, 품질 메트릭 |
| 레벨 5: 최적화 | 지속적 개선, 커뮤니티 기여, 재사용 최대화 |

## 정리

도메인 전문가와의 협업은 온톨로지 품질을 결정하는 가장 중요한 요소입니다.

- **지식 획득** : 반구조화 인터뷰, 카드 소팅, 프로토타입 검토, 시나리오 워크스루
- **실패 패턴** : 범위 미정의, 전문가 부재, 위원회 설계, 구현 종속, 완벽주의
- **피로도 관리** : 세션 단축, 피드백 가시화, 비기술적 표현, 인정
- 완벽한 온톨로지를 목표로 하지 말고, 사용 가능한 온톨로지를 배포한 뒤 개선합니다
