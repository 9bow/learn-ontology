Naive RAG는 벡터 유사도로 관련 청크를 검색하지만, 검색된 청크가 답변에 필요한 모든 사실을 담고 있다는 보장이 없다. Ontology-guided RAG는 도메인 온톨로지를 검색·생성 파이프라인에 통합해 정확성을 높이고 환각을 줄이는 패턴이다.

```text
온톨로지 기반 환각 감소 메커니즘

일반 RAG (환각 위험)
  사용자 질의
      │
      ▼ 벡터 유사도
  청크 검색 (관련 없는 청크 혼입 가능)
      │
      ▼
  LLM 자유 생성 ──▶ 환각 발생 가능
      │
      ▼
  답변 (출처 불투명)

Ontology-guided RAG (환각 억제)
  사용자 질의
      │
      ├─▶ 온톨로지 쿼리 확장 (서브타입 자동 포함)
      │         "당뇨병" → 1형/2형/임신성 당뇨 모두 검색
      │
      ├─▶ KG 경로 탐색 (다중 홉)
      │         Drug → contraindicated_with → ContrastMedia
      │
      ▼
  구조화 컨텍스트 주입 (사실·출처 명시)
      │
      ▼
  LLM 제약 생성 (온톨로지 범위 내)
      │
      ▼
  답변 검증 (SPARQL ASK)
      │
      ├─▶ KG 반증 없음 → 답변 반환
      └─▶ KG 반증 발견 → 수정·불확실성 표시
```

## 왜 온톨로지가 RAG를 개선하는가

**환각 감소 메커니즘:**
온톨로지는 도메인에서 허용되는 개념·관계·제약 조건을 명시적으로 정의한다. LLM이 생성한 답변을 온톨로지로 검증하면 존재하지 않는 관계나 타입 위반을 탐지할 수 있다.

**쿼리 확장:**
사용자가 "당뇨병 치료제"를 질문할 때, 당뇨병의 서브타입(1형, 2형, 임신성 당뇨)까지 온톨로지의 클래스 계층으로 자동 확장해 더 포괄적인 검색이 가능하다.

**다중 홉 추론:**
벡터 유사도는 직접 연결된 정보만 검색한다. KG 경로 탐색은 A→B→C처럼 여러 단계를 거쳐 연결된 사실을 추적할 수 있다.

## 패턴 1: 온톨로지 기반 쿼리 확장

검색 전 단계에서 온톨로지를 활용해 사용자 질의를 풍부하게 만든다.

**구현 방식:**
1. 질의에서 엔티티·개념을 추출한다.
2. 온톨로지에서 해당 개념의 동의어, 상위/하위 클래스, 관련 프로퍼티를 가져온다.
3. 확장된 용어로 벡터 검색 또는 키워드 검색을 수행한다.

```sparql
# SNOMED CT에서 "심근경색"의 하위 개념 조회
SELECT ?subtype ?label WHERE {
  ?subtype rdfs:subClassOf* snomedct:22298006 ;
           rdfs:label ?label .
}
```

이 SPARQL 결과를 질의 확장에 사용하면 "STEMI", "NSTEMI", "급성 심근경색" 등 관련 문서를 모두 검색에 포함할 수 있다.

## 패턴 2: KG 경로 탐색 기반 다중 홉 검색

벡터 검색만으로는 다중 홉 질문에 답하기 어렵다. KG 경로 탐색과 벡터 검색을 결합하면 이를 해결할 수 있다.

**예시 질문:** "메트포르민을 복용 중인 2형 당뇨 환자에게 금기인 조영제는?"

**KG 경로 탐색 흐름:**
1. "메트포르민" → 온톨로지에서 `DrugClass: Biguanide` 식별
2. "2형 당뇨" → `Condition: Type2DiabetesMellitus` 식별
3. KG에서 `Biguanide` ← `contraindicated_with` → `IodineContrastMedia` 경로 탐색
4. 탐색된 관계를 컨텍스트로 LLM에 제공

단순 벡터 검색은 "메트포르민"과 "조영제"가 같은 문서에 언급된 청크를 찾아야 하지만, KG 탐색은 관계 자체를 직접 조회한다.

## 패턴 3: 구조화 컨텍스트 주입

검색된 청크를 그대로 주입하는 대신, 온톨로지 구조로 정제된 사실을 컨텍스트로 제공한다.

**비교:**

Naive RAG 컨텍스트:
```
"...메트포르민은 2형 당뇨 치료에 널리 사용되는 약물로, 신기능 저하 환자에서는 주의가 필요하다. 조영제를 사용하는 검사 전에 일시적으로 중단을 권고하기도 한다..."
```

Ontology-guided RAG 컨텍스트:
```
사실 (출처: DrugBank DB00331):
- 메트포르민 [약물 클래스: Biguanide]
- 금기: 사구체 여과율(GFR) < 30 mL/min/1.73m²
- 주의: 요오드 조영제 투여 48시간 전후 중단 권고
관련 가이드라인: ADA 2024, FDA 라벨 2023
```

구조화된 사실 기반 컨텍스트는 LLM이 정확한 정보를 선택적으로 활용하게 돕고, 출처 추적이 용이하다.

## 패턴 4: 온톨로지 기반 답변 검증

LLM이 생성한 답변을 온톨로지로 사후 검증한다.

**구현:**
1. LLM 답변에서 주장(claim)을 추출한다.
2. 각 주장을 SPARQL ASK 쿼리로 KG에 검증한다.
3. KG에서 반증되는 주장이 있으면 답변을 수정하거나 불확실성을 표시한다.

```sparql
# 답변 "메트포르민은 1형 당뇨에 금기다" 검증
ASK {
  drugbank:DB00331drug:contraindicatedFor condition:Type1DiabetesMellitus .
}
```

이 검증이 FALSE를 반환하면 해당 주장을 답변에서 제거하거나 "확인 필요"로 표시한다.

## 구현 아키텍처

```
사용자 질의
    ↓
[엔티티 인식 + 온톨로지 매핑]
    ↓
[SPARQL/KG 경로 탐색] ←→ [벡터 유사도 검색]
    ↓              ↓
[KG 사실]    [관련 청크]
    ↓              ↓
    [컨텍스트 병합·랭킹]
          ↓
    [LLM 답변 생성]
          ↓
    [온톨로지 기반 검증]
          ↓
    최종 답변 (출처 포함)
```

## 주요 트레이드오프

| 항목 | 장점 | 단점 |
|------|------|------|
| 쿼리 확장 | 재현율 향상 | 관련 없는 결과 증가 가능 |
| KG 경로 탐색 | 다중 홉 정확도 | KG 구축·유지 비용 |
| 구조화 컨텍스트 | 환각 감소 | 비정형 지식 포착 한계 |
| 답변 검증 | 정밀도 향상 | 지연 시간 증가 |

## 실무 적용 시 고려 사항

- **KG 품질이 전제:** 온톨로지와 KG가 정확하지 않으면 오히려 검색 품질이 저하된다. KG 구축과 유지에 충분한 투자가 필요하다.
- **커버리지 한계:** KG에 없는 지식은 탐색할 수 없다. 벡터 검색과 KG 탐색의 결합이 커버리지를 보완한다.
- **지연 시간:** SPARQL 쿼리 + 벡터 검색 + LLM 생성 + 검증의 순차 실행은 응답 시간을 늘린다. 비동기 병렬 실행과 캐싱으로 완화한다.
- **도메인 특화:** 의료, 법률, 금융처럼 정확성이 중요한 도메인에서 ROI가 크다. 일반 도메인에서는 복잡도 대비 이점이 제한될 수 있다.
