LLM 에이전트는 도구(tool)를 호출해 외부 세계와 상호작용한다. 웹 검색, 데이터베이스 조회, API 호출, 코드 실행 등이 도구의 예다. 에이전트가 강력해질수록 잘못된 도구 호출의 결과도 심각해진다. 온톨로지는 에이전트가 도구를 이해하고, 올바른 도구를 선택하며, 실행 결과를 검증하는 의미 계층(semantic layer)을 제공한다.

## 왜 에이전트에게 온톨로지가 필요한가

순수 LLM 에이전트의 도구 선택은 도구 설명 텍스트의 의미적 유사도에 의존한다. 이 방식은 세 가지 문제가 있다.

**1. 도구 오용:** "환자 데이터 삭제" 도구와 "환자 데이터 보관 처리" 도구가 설명이 유사하면 LLM이 잘못 선택할 수 있다. 온톨로지는 도구의 사전 조건(precondition), 사후 조건(postcondition), 권한 요구 사항을 명시적으로 정의한다.

**2. 도메인 제약 무시:** LLM은 비즈니스 규칙이나 규제 제약을 모를 수 있다. 예: "FDA 승인 전 약물은 임상 시험 외 처방 불가". 온톨로지는 이런 제약을 공리로 인코딩해 에이전트가 위반하기 전에 탐지한다.

**3. 추론 불투명:** 에이전트가 왜 그 도구를 선택했는지 설명하기 어렵다. 온톨로지 기반 의사결정은 근거(ontology path)를 추적 가능하다.

## Palantir AIP: Ontology-Centric Agent Platform

Palantir AIP는 Foundry의 Ontology 위에 LLM 에이전트를 구축하는 플랫폼이다. 6섹션에서 소개한 Objects·Links·Actions 메타모델이 에이전트 컨텍스트에서 어떻게 작동하는지 살펴보자.

**Objects as Agent Context:**
에이전트가 작업할 때 관련 Object(환자, 주문, 자산 등)를 온톨로지에서 조회해 컨텍스트로 제공한다. LLM은 비정형 텍스트 대신 구조화된 온톨로지 객체를 입력받아 더 정확하게 추론한다.

**Actions as Tool Definitions:**
에이전트가 호출할 수 있는 도구는 Ontology Actions로 정의된다. 각 Action은 다음을 포함한다:
- 입력 파라미터 타입 (Object 타입 기반)
- 검증 규칙 (파라미터 값 제약)
- 권한 요구 (어떤 역할이 실행 가능한가)
- 사이드 이펙트 (어떤 Object가 변경되는가)

LLM은 Action 메타데이터를 보고 적합한 Action을 선택하며, 선택 전에 검증 규칙이 자동으로 실행된다.

**Safety Layer:**
AIP는 에이전트가 실제로 Action을 실행하기 전에 사람의 검토를 요구하는 "Human-in-the-loop" 단계를 온톨로지 레벨에서 설정할 수 있다. 특정 Object 타입(예: "환자 치료 계획")이 포함된 Action은 반드시 의료 전문가가 승인해야 실행된다.

**사례: NHS 임상 운영:**
NHS가 Palantir Foundry를 도입한 영국 NHS FDP(Federated Data Platform)에서 병상 배정, 수술 일정 조정 같은 운영 의사결정을 AIP 에이전트가 지원한다. 에이전트가 제안하는 병상 배정은 온톨로지의 환자 분류 규칙과 병동 역량 제약을 자동으로 검증한 후 담당자에게 승인 요청을 보낸다.

## Neo4j GenAI 스택

Neo4j는 LLM + KG 통합을 위한 GenAI 스택을 제공한다. 핵심은 그래프를 에이전트의 장기 메모리와 추론 컨텍스트로 활용하는 것이다.

**GraphRAG + Vector Index:**
Neo4j는 벡터 인덱스를 내장해 그래프 탐색과 벡터 유사도 검색을 단일 Cypher 쿼리로 결합할 수 있다.

```cypher
// 그래프 관계 탐색 + 벡터 유사도 결합
MATCH (doc:Document)-[:MENTIONS]->(entity:Entity {name: $entityName})
CALL db.index.vector.queryNodes('doc_embeddings', 5, $queryEmbedding)
YIELD node, score
RETURN node.content, score
ORDER BY score DESC
```

**에이전트 메모리 그래프:**
대화 이력, 사용자 선호도, 이전 추론 결과를 KG 노드·엣지로 저장한다. 에이전트가 새 질문을 받으면 메모리 그래프를 조회해 이전 컨텍스트를 복원한다. Reflexion(Shinn et al. 2023)처럼 이전 실패를 언어적으로 반성해 KG에 저장하면 동일 오류를 반복하지 않는다.

**LangChain/LlamaIndex 통합:**
Neo4j는 LangChain의 `Neo4jGraph`, `GraphCypherQAChain`과 LlamaIndex의 `KnowledgeGraphIndex`를 공식 지원한다. 에이전트 프레임워크에서 KG를 도구로 등록해 자동 호출하는 패턴이 표준화되고 있다.

## 온톨로지 기반 도구 선택 메커니즘

에이전트가 여러 도구 중 올바른 것을 선택하려면 도구의 의미를 이해해야 한다. 온톨로지 기반 도구 카탈로그는 이를 체계화한다.

**도구 온톨로지 예시 (Turtle):**
```turtle
:SearchPatientRecords a :Tool ;
    :name "환자 기록 검색" ;
    :requiresPermission :PatientDataRead ;
    :inputType :PatientIdentifier ;
    :outputType :PatientRecord ;
    :sideEffect :None ;
    :applicableContext :ClinicalDecisionSupport .

:UpdateTreatmentPlan a :Tool ;
    :name "치료 계획 수정" ;
    :requiresPermission :TreatmentPlanWrite ;
    :requiresApproval :AttendingPhysician ;
    :inputType :TreatmentPlanDraft ;
    :sideEffect :ModifiesPatientRecord .
```

에이전트는 현재 컨텍스트(사용자 권한, 작업 유형)와 도구 온톨로지를 매칭해 호출 가능한 도구 집합을 좁힌 뒤 LLM으로 최종 선택한다. 권한 없는 도구는 후보 목록에서 제외되므로 LLM이 환각으로 잘못된 도구를 선택해도 실행되지 않는다.

## 현실적 한계와 고려 사항

**온톨로지 구축 선행 투자:** 에이전트에 활용할 수 있는 수준의 도메인 온톨로지는 구축에 상당한 시간이 필요하다. 범용 에이전트보다 특정 도메인에 집중한 에이전트부터 시작하는 것이 현실적이다.

**동적 도구 환경:** 도구가 자주 추가·변경되는 환경에서는 온톨로지 유지 비용이 발생한다. 온톨로지 변경이 에이전트 동작에 미치는 영향을 테스트하는 파이프라인이 필요하다.

**LLM과 온톨로지의 불일치:** LLM은 온톨로지의 정밀한 타입 체계를 이해하지 못할 수 있다. 온톨로지 정보를 프롬프트로 변환하는 "온톨로지 직렬화" 품질이 에이전트 성능에 영향을 미친다.

**감사 추적(Audit Trail):** 온톨로지 기반 에이전트는 각 결정의 근거(사용된 온톨로지 경로, 검증된 제약, 승인 이력)를 기록할 수 있다. 규제 산업에서 에이전트 행동의 감사 추적을 요구할 때 이 기록이 핵심 증거가 된다.
