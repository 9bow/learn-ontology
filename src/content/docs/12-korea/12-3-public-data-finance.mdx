## 한국 공공데이터와 개방 현황

한국 정부는 2013년 공공데이터법 제정 이후 **data.go.kr(공공데이터 포털)** 을 중심으로 대규모 데이터 개방을 추진했습니다. 2024년 기준으로 수만 건의 데이터셋이 개방되어 있으며, API 형태로 제공되는 실시간 데이터도 포함됩니다.

그러나 공공데이터의 품질과 상호운용성은 여전히 과제입니다. 기관마다 다른 분류 체계, 비표준 코드, 불일치하는 필드명이 데이터 활용을 어렵게 합니다. 예를 들어 "시도 코드"를 어떤 기관은 숫자 코드(11=서울)로, 다른 기관은 문자 코드(SEL=서울)로 사용합니다.

이 문제를 해결하기 위해 행정안전부는 공공데이터 표준화 지침을 마련했고, 공통 분류 체계와 코드 표준을 정의했습니다. 이 표준 코드 체계는 온톨로지의 SKOS ConceptScheme으로 표현하기에 적합한 구조입니다.

## 공공데이터 메타데이터와 DCAT

공공데이터 포털의 데이터셋 메타데이터는 국제 표준인 **DCAT(Data Catalog Vocabulary)** 와 호환되도록 발전하고 있습니다. DCAT는 W3C 권고 어휘로, 데이터 카탈로그를 RDF로 기술하는 표준입니다.

DCAT 기반 메타데이터 예시:

```turtle
<https://www.data.go.kr/data/15000581/openapi.do>
    a dcat:Dataset ;
    dct:title "건강보험심사평가원_의약품 정보 서비스"@ko ;
    dct:publisher <https://www.hira.or.kr/> ;
    dct:issued "2015-01-01"^^xsd:date ;
    dct:modified "2024-03-15"^^xsd:date ;
    dcat:keyword "의약품"@ko, "약품정보"@ko ;
    dct:license <https://www.data.go.kr/license> .
```

DCAT로 메타데이터를 표준화하면 유럽의 EU Open Data Portal 같은 글로벌 카탈로그와 연계할 수 있고, SPARQL로 여러 기관의 데이터셋을 통합 검색할 수 있습니다.

## 보건의료 공공 온톨로지

한국에서 온톨로지가 가장 적극적으로 적용된 공공 데이터 영역 중 하나는 보건의료입니다.

**건강보험심사평가원(HIRA):** 의약품 정보, 진료 코드(EDI 코드), 질병 분류(KCD, 한국표준질병사인분류)를 표준화해 API로 제공합니다. KCD는 WHO의 ICD-10을 한국 실정에 맞게 수정한 분류 체계로, SNOMED CT와 매핑 연구가 진행 중입니다.

**국립중앙의료원 — 의료 데이터 표준:** HL7 FHIR를 도입해 전자의무기록(EMR) 데이터의 상호운용성을 높이는 작업이 진행 중입니다. FHIR 리소스는 SNOMED CT, LOINC 같은 의료 온톨로지를 코딩 시스템으로 참조합니다.

## 금융권 데이터 표준화

한국 금융 분야의 데이터 표준화는 여러 기관이 관여합니다.

**금융위원회·금융감독원:** 금융 보고 체계(XBRL 기반)를 통해 상장사의 재무 보고를 표준화합니다. XBRL(eXtensible Business Reporting Language)은 재무 데이터를 구조화하는 XML 기반 언어로, 온톨로지와 유사한 개념 정의 구조를 가집니다.

**금융보안원:** 금융 분야 사이버 보안과 데이터 보호 가이드라인을 제시합니다. 금융 데이터의 분류 체계(개인정보, 중요 정보, 일반 정보)를 정의하는 것은 온톨로지적 접근과 연결됩니다.

**은행연합회·금융투자협회:** 업권별 공통 데이터 표준을 마련합니다. 예를 들어 금융투자협회는 증권사 간 공통 코드 체계(종목 코드, 거래 유형 코드)를 관리합니다.

## 오픈 파이낸스와 MyData

2021년 도입된 **마이데이터(MyData)** 제도는 금융 분야 지식 그래프 관점에서 흥미로운 사례입니다. 마이데이터는 소비자가 자신의 금융 데이터(은행, 카드, 보험, 증권)를 한곳에서 조회하고 제3자 서비스에 제공할 수 있도록 허용하는 제도입니다.

마이데이터 표준 API는 금융보안원이 정의하며, 모든 참여 기관이 동일한 API 스펙을 따라야 합니다. 이 과정에서 계좌, 거래, 상품을 표현하는 공통 어휘가 정의됩니다. 이는 FIBO 같은 금융 온톨로지의 한국판 역할을 합니다.

마이데이터를 통해 수집된 데이터를 지식 그래프로 구성하면 개인 재무 분석, 맞춤형 금융 상품 추천, 이상 거래 탐지 등의 서비스가 가능합니다.

## 도입 장벽과 현실

국내 금융권의 온톨로지·지식 그래프 도입은 선도적인 일부 기관을 제외하면 아직 초기 단계입니다. 주요 장벽은 다음과 같습니다.

**레거시 시스템:** 수십 년된 코볼(COBOL) 기반 코어뱅킹 시스템은 데이터 추출과 연동이 어렵습니다.

**전문 인력 부족:** OWL, SPARQL, 온톨로지 설계 경험을 가진 전문가가 국내 금융권에 드뭅니다.

**규제 불확실성:** 금융 데이터를 외부 플랫폼(클라우드 기반 트리플 스토어)에 저장하는 것에 대한 규제 해석이 명확하지 않은 경우가 있습니다.

**성과 측정의 어려움:** 온톨로지 도입의 ROI를 단기간에 측정하기 어려워 경영진 설득이 쉽지 않습니다.

## 정리

한국의 공공데이터와 금융권은 표준화와 상호운용성 요구로 인해 온톨로지 적용 가능성이 높은 영역입니다. 마이데이터, FHIR, DCAT 같은 국제 표준의 국내 도입이 가속화되면서 온톨로지 기반 접근법의 현실적 필요성도 높아지고 있습니다. 다음 챕터에서는 한국어 도메인 특유의 언어적 과제를 살펴봅니다.
