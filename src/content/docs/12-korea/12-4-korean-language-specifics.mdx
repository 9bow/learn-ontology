## 교착어로서의 한국어

한국어는 **교착어(agglutinative language)** 입니다. 어근에 다양한 접사(조사, 어미)가 붙어 문법적 기능이 결정됩니다. 이 특성은 온톨로지와 지식 그래프 구축에 여러 방면에서 영향을 미칩니다.

영어에서 "Samsung"은 형태가 변하지 않습니다. 그러나 한국어에서는 같은 개체가 문장 내 위치에 따라 "삼성이", "삼성을", "삼성의", "삼성에서", "삼성으로" 등으로 변형됩니다. 개체명 인식(NER) 모델은 이 모든 변형을 "삼성"이라는 동일 개체로 인식해야 합니다.

```
원문: 삼성전자가 새로운 반도체를 개발했다고 삼성전자의 관계자가 밝혔다.
개체: [삼성전자]가 → Entity: 삼성전자
개체: [삼성전자]의 → Entity: 삼성전자 (동일 개체)
```

형태소 분석기(KoNLPy, Mecab-ko, 은전한닢 등)가 이 분리를 수행하지만, 분석기마다 세분화 기준이 달라 일관성 문제가 생깁니다.

## 온톨로지 레이블과 형태소

OWL 온톨로지에서 클래스와 속성에는 `rdfs:label`로 자연어 레이블을 부여합니다. 한국어 레이블을 정의할 때 형태소 기준으로 정규형을 결정해야 합니다.

**권장 원칙:**
- 명사형 레이블을 사용합니다. "개발하다" 대신 "개발", "제조하다" 대신 "제조"를 레이블로 사용합니다.
- 복합명사는 공백 없이 붙여 씁니다(한국어 어문 규정 따름). "지식그래프"보다 "지식 그래프"가 표준이지만, 도메인 관행을 우선합니다.
- `skos:altLabel`로 변형 표현을 모두 등록합니다.

```turtle
:ManufacturingProcess a owl:Class ;
    rdfs:label "제조 공정"@ko ;
    rdfs:label "Manufacturing Process"@en ;
    skos:altLabel "생산 공정"@ko ;
    skos:altLabel "제조공정"@ko .
```

## 존칭과 경어 체계의 문제

한국어는 **존칭 체계**가 매우 발달해 있습니다. 같은 의미의 문장도 화자·청자·주어의 관계에 따라 형태가 크게 달라집니다.

```
평어: 이 회사가 제품을 만든다.
경어: 이 회사가 제품을 만듭니다.
높임: 회장님께서 말씀하셨다.
```

관계 추출 파이프라인에서 "X가 Y를 말했다"와 "X께서 Y를 말씀하셨다"는 같은 관계(`:said`)를 표현하지만, 형태가 다릅니다. 존칭 표현을 인식하지 못하는 모델은 이 두 패턴을 같은 관계로 매핑하지 못합니다.

실용적 해결책은 다음과 같습니다:
- **정규화 전처리:** 존칭 표현을 평어로 정규화한 뒤 관계 추출을 적용합니다.
- **존칭 인식 레이어:** 발화자-대상 간 사회적 관계를 추가 속성으로 기록합니다(예: `:socialRegister :Honorific`). 이는 인물 관계 온톨로지에서 유용합니다.

## 복합 고유명사의 경계 모호성

한국어 고유명사는 **경계가 불분명한 경우**가 많습니다.

```
삼성전자주식회사 → [삼성전자] + [주식회사]? 또는 [삼성] + [전자주식회사]?
서울특별시강남구역삼동 → 행정 계층을 어떻게 분리?
현대자동차그룹글로벌비즈니스센터 → ?
```

온톨로지에서 이 문제를 처리하는 방법입니다.

**정규 표기 우선:** LEI(Legal Entity Identifier)에 등록된 법인명을 정규 표기로 사용하고, 다른 표기는 `skos:altLabel`로 등록합니다.

**행정 계층 온톨로지:** 행정구역 코드(행정안전부 표준 코드)를 개체의 식별자로 사용하고, 텍스트 표현은 레이블로 분리합니다.

```turtle
:KR-11-680 a :AdministrativeDistrict ;
    rdfs:label "강남구"@ko ;
    skos:altLabel "서울특별시 강남구"@ko ;
    :administrativeCode "11680" ;
    :partOf :KR-11 .  # 서울특별시
```

## 동음이의어와 다의어

한국어에는 동음이의어와 다의어가 많습니다. 온톨로지에서 이를 처리하지 않으면 추론 오류가 발생합니다.

**동음이의어 예시:**
- "배" — 신체 부위(복부), 과일(배나무), 교통수단(선박)
- "차" — 음료(차茶), 교통수단(자동차), 차이(差)

**다의어 예시:**
- "장관" — 정부 각료(長官), 훌륭한 경치(壯觀)

온톨로지 설계 원칙:
1. 클래스와 속성에 URN 기반 고유 식별자를 사용합니다. 동일한 레이블이 있어도 URI가 다르면 다른 개체입니다.
2. `skos:note`나 `skos:scopeNote`로 사용 범위를 명시합니다.
3. 인스턴스 생성 시 타입(rdf:type)을 명시적으로 지정합니다.

```turtle
:Bae_Fruit a :Fruit ;
    rdfs:label "배"@ko ;
    skos:scopeNote "배나무(Pyrus) 열매"@ko .

:Bae_Ship a :WaterVehicle ;
    rdfs:label "배"@ko ;
    skos:scopeNote "수상 교통수단"@ko .
```

## 신조어와 외래어

한국어는 신조어 생성 속도가 빠르고, 영어 외래어가 음차되어 사용됩니다.

**음차 표기의 불일치:**
- "Knowledge Graph" → "지식 그래프" / "놀리지 그래프" / "KG"
- "Ontology" → "온톨로지" / "존재론" / "본체론"

온톨로지에서는 `skos:altLabel`로 모든 표기를 등록하고, `rdfs:label`의 정규 표기를 명시적으로 결정해야 합니다. 국립국어원의 외래어 표기법을 기준으로 삼는 것이 좋습니다.

**신조어 반영:** 온톨로지는 정적 문서가 아니라 지속적으로 갱신되어야 합니다. 새로운 도메인 용어는 버전 관리와 함께 추가하고, 기존 개념과의 관계를 명시합니다.

## 다국어 레이블 관리

한국 기업이나 기관이 운영하는 온톨로지는 한국어와 영어 레이블을 함께 관리해야 합니다. 글로벌 파트너나 국제 표준과의 연계를 위해서입니다.

```turtle
:KoreanCentralBank a fibo-be:CentralBank ;
    rdfs:label "한국은행"@ko ;
    rdfs:label "Bank of Korea"@en ;
    skos:altLabel "BOK"@en ;
    owl:sameAs <http://www.wikidata.org/entity/Q487081> .
```

`owl:sameAs`로 Wikidata 항목과 연결하면 다국어 레이블을 자동으로 상속할 수 있습니다.
