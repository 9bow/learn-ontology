## OWL 2 프로파일 트레이드오프

```text
표현력
  ▲
  │  OWL 2 Full ──────────────────────────── 결정 불가능(undecidable)
  │       (메타모델링 허용, 추론 엔진 없음)
  │
  │  OWL 2 DL ─────────────────────────────  2-NEXPTIME
  │       (SROIQ, 완전 추론, 소규모 데이터)  HermiT · Pellet
  │
  │  OWL 2 EL ─────────────────────────────  P-time ★ 생의학
  │       (교집합·존재제한, 대형 분류)        ELK · Snorocket
  │       SNOMED CT · Gene Ontology
  │
  │  OWL 2 RL ─────────────────────────────  P-time ★ 대규모 KG
  │       (규칙→Datalog 변환)                RDFox · GraphDB
  │       엔터프라이즈 RDF 추론
  │
  │  OWL 2 QL ─────────────────────────────  NLogSpace ★ DB 연동
  │       (SPARQL→SQL 재작성, OBDA)           Ontop · Stardog
  │       기존 관계형 DB에 온톨로지 뷰
  │
  │  RDFS ──────────────────────────────────  선형 ★ 대규모 웹
  │       (경량 스키마, schema.org, FOAF)
  │
  └─────────────────────────────────────────────────────────▶
                                                     추론 효율성
```

## 왜 프로파일이 존재하는가

OWL 2는 매우 표현력이 높은 언어입니다. 그러나 표현력과 추론 복잡도는 반비례합니다. OWL 2 DL의 완전한 추론은 2-NEXPTIME(이중 비결정론적 지수 시간)입니다. 수백만 개의 개체를 가진 기업 데이터에 이 추론을 전체 적용하면 실용적 성능이 나오지 않습니다.

이 문제를 해결하기 위해 W3C OWL 2 표준은 **프로파일(profiles)** — 표현력의 특정 부분집합 — 을 정의했습니다. 각 프로파일은 특정 사용 사례에 맞게 표현력을 제한하고 대신 효율적인 추론을 보장합니다.

## OWL 1의 세 표현력 수준

### OWL Lite

OWL DL의 단순화 버전입니다. 카디널리티는 0 또는 1만 허용하고, 열거 클래스(`owl:oneOf`)는 지원하지 않습니다. 실제로는 OWL DL과 구분이 모호해 사용이 줄었습니다.

### OWL DL (Description Logic)

OWL의 핵심 프로파일입니다. 기술 논리(DL) SROIQ에 대응합니다. 추론이 **결정 가능(decidable)** 하고 완전(complete)합니다. 다만 복잡도가 높습니다.

주요 제약: 클래스와 개체가 다른 이름공간을 가져야 합니다(메타모델링 불가).

### OWL Full

OWL DL + 메타모델링(클래스가 개체이기도 한 것)을 허용합니다. 표현력이 최고지만 **추론이 결정 불가능(undecidable)** 합니다. "클래스의 클래스", "속성의 속성" 같은 표현이 가능합니다.

실용적 시스템에서 OWL Full을 완전 추론하는 엔진은 존재하지 않습니다.

## OWL 2 프로파일: EL, QL, RL

OWL 2는 실용적 추론을 위해 세 프로파일을 추가했습니다. 각 프로파일은 OWL 2 DL의 부분집합입니다.

### OWL 2 EL (Existential Language)

**설계 목적:** 대규모 생의학 온톨로지 (SNOMED CT, Gene Ontology, NCI Thesaurus)

**추론 복잡도:** P-time (다항 시간) — 수백만 클래스도 분 단위 분류 가능

**지원 기능:**
- 클래스 교집합 (`owl:intersectionOf`)
- 존재 제한 (`owl:someValuesFrom`)
- 자기 자신 제한 (`owl:Self`)
- 추이적 속성
- 기능적 속성 (Data Property만)

**미지원 기능:**
- 전칭 제한 (`owl:allValuesFrom`)
- 불교차 (`owl:disjointWith`) — 일부 제한적 지원
- 카디널리티 제한
- 합집합 (`owl:unionOf`)

```turtle
# OWL 2 EL 호환 예시
:DiseasedCell owl:equivalentClass [
    owl:intersectionOf (
        :Cell
        [ owl:onProperty :hasPathologicalProcess ;
          owl:someValuesFrom :PathologicalProcess ]
    )
] .
# SNOMED CT의 전형적인 패턴
```

**추천 추론 엔진:** ELK (Apache License 2.0), Snorocket

### OWL 2 QL (Query Language)

**설계 목적:** 대규모 RDF 데이터셋에서 SPARQL 쿼리 재작성(query rewriting)

**추론 복잡도:** NLogSpace (데이터 복잡도) — SPARQL 쿼리를 SQL로 변환해 DB에서 직접 실행 가능

**핵심 아이디어:** OWL 2 QL 온톨로지를 이용해 SPARQL 쿼리를 확장된 SQL/SPARQL 쿼리로 **재작성**합니다. 이를 통해 기존 관계형 DB 데이터에 온톨로지 추론을 적용할 수 있습니다 (Ontology-Based Data Access, OBDA).

**지원 기능:**
- 클래스 계층 (subClassOf)
- 속성 계층 (subPropertyOf)
- 도메인·범위
- 역속성 (inverseOf)
- 교집합 (제한적)

**미지원 기능:**
- 존재 제한의 우측 (range side)
- 추이적 속성
- 카디널리티

```sparql
-- OWL 2 QL + OBDA 시나리오
-- 온톨로지: Employee rdfs:subClassOf Person
-- 실제 데이터는 MySQL의 'employees' 테이블에 있음
-- Ontop (OBDA 도구)이 SPARQL을 SQL로 재작성

SELECT ?name WHERE {
  ?p rdf:type :Person ;   -- Employee도 Person으로 추론
     :hasName ?name .
}
-- 재작성된 SQL: SELECT name FROM employees WHERE ...
```

**추천 도구:** Ontop (SPARQL-to-SQL 변환), Stardog Virtual Graphs

### OWL 2 RL (Rule Language)

**설계 목적:** 규칙 기반 추론 엔진 (RIF, Datalog)과의 통합, 대규모 데이터에 전향 연쇄 규칙 적용

**추론 복잡도:** P-time (데이터 복잡도)

**핵심 아이디어:** OWL 2 RL의 공리들은 **Datalog 규칙**으로 변환할 수 있습니다. 이를 통해 대규모 RDF 데이터에 규칙 기반 추론을 적용합니다.

**지원 기능:**
- 클래스 계층, 속성 계층
- 전칭 제한 (`allValuesFrom`) — 좌측에서만
- 존재 제한 — 좌측에서만
- 추이적 속성
- 대칭·반대칭 속성
- 불교차 클래스

**미지원 기능:**
- 존재 제한의 우측 (새 개체 생성 불가)
- 카디널리티
- 열거 클래스

```turtle
# OWL 2 RL 호환 예시: 규칙으로 변환 가능한 공리
:Physician rdfs:subClassOf :Person .         # RL 규칙: type(X, Physician) → type(X, Person)
:prescribes rdfs:domain :Physician .         # RL 규칙: prescribes(X,Y) → type(X, Physician)
:prescribes rdfs:range :Drug .               # RL 규칙: prescribes(X,Y) → type(Y, Drug)
:hasParent owl:inverseOf :hasChild .         # RL 규칙: hasParent(X,Y) ↔ hasChild(Y,X)
:hasAncestor a owl:TransitiveProperty .      # RL 규칙: hasAncestor(X,Y) ∧ hasAncestor(Y,Z) → hasAncestor(X,Z)
```

**추천 추론 엔진:** RDFox, GraphDB (RL 모드), Stardog

## 프로파일 종합 비교

| 항목 | OWL 2 DL | OWL 2 EL | OWL 2 QL | OWL 2 RL |
|------|---------|---------|---------|---------|
| 설계 목적 | 완전한 추론 | 대형 분류 | DB 쿼리 재작성 | 규칙 기반 추론 |
| 추론 복잡도 | 2-NEXPTIME | P-time | NLogSpace | P-time |
| 클래스 교집합 | ✅ | ✅ | ✅ (제한) | ✅ |
| 클래스 합집합 | ✅ | ❌ | ❌ | ✅ (제한) |
| 존재 제한 (우측) | ✅ | ✅ | ❌ | ❌ |
| 전칭 제한 | ✅ | ❌ | ❌ | ✅ (좌측) |
| 카디널리티 | ✅ | ❌ | ❌ | ❌ |
| 추이적 속성 | ✅ | ✅ | ❌ | ✅ |
| 역속성 | ✅ | ❌ | ✅ | ✅ |
| 대표 사례 | 범용 | SNOMED CT, GO | OBDA | 대규모 KG |

## 선택 가이드

**수백만 클래스의 생의학 온톨로지** — OWL 2 EL + ELK

**기존 관계형 DB에 온톨로지 뷰 씌우기** — OWL 2 QL + Ontop

**엔터프라이즈 RDF 데이터에 규칙 추론** — OWL 2 RL + RDFox

**완전한 추론 필요, 데이터 규모 작음** — OWL 2 DL + HermiT/Pellet

**추론 불필요, 스키마만** — RDFS

프로파일 선택은 온톨로지 설계 초기에 결정해야 합니다. 나중에 프로파일을 바꾸면 온톨로지 전체를 리팩터링해야 할 수 있습니다. 다음 챕터에서는 OWA 환경의 검증 표준인 SHACL을 살펴봅니다.
