## SHACL이 필요한 이유

OWL 추론은 "이 데이터에서 무엇을 도출할 수 있는가?"를 다룹니다. 그러나 실무 데이터 파이프라인에서는 다른 질문이 더 중요할 때가 많습니다. "이 데이터가 우리가 정한 형식을 따르는가?"

예를 들어 ETL 파이프라인에서 데이터가 들어올 때, 다음을 확인해야 합니다.

- 모든 직원에게 부서 정보가 있는가?
- 이메일 주소 형식이 올바른가?
- 나이가 0 이상 150 이하인가?
- 조직 코드가 허용된 값 목록에 있는가?

OWL은 열린 세계 가정(OWA) 때문에 이런 검증에 적합하지 않습니다. **SHACL(Shapes Constraint Language)** 은 이 공백을 채웁니다. W3C 권고안(2017)으로, RDF 데이터를 닫힌 세계 가정 스타일로 검증합니다.

## SHACL의 기본 구조

SHACL은 **Shape(형태)** 라는 개념으로 제약을 정의합니다. Shape는 "어떤 RDF 노드가 어떤 조건을 만족해야 하는가"를 기술합니다.

두 가지 Shape 타입이 있습니다.

- **NodeShape** — 노드 자체에 대한 제약 (어떤 클래스의 인스턴스여야 하는가 등)
- **PropertyShape** — 노드가 가진 특정 속성에 대한 제약

---

## NodeShape 예시

```turtle
@prefix sh:   <http://www.w3.org/ns/shacl#> .
@prefix rdf:  <http://www.w3.org/1999/02/22-rdf-syntax-ns#> .
@prefix xsd:  <http://www.w3.org/2001/XMLSchema#> .
@prefix :     <http://example.org/> .

# NodeShape: Person 클래스의 모든 인스턴스에 적용
:PersonShape
    rdf:type sh:NodeShape ;
    sh:targetClass :Person ;       # :Person의 모든 인스턴스에 적용
    sh:property [
        sh:path :hasName ;
        sh:minCount 1 ;            # 반드시 하나 이상의 이름이 있어야 함
        sh:maxCount 1 ;            # 이름은 하나여야 함
        sh:datatype xsd:string ;   # 문자열 타입
    ] ;
    sh:property [
        sh:path :hasEmail ;
        sh:minCount 1 ;
        sh:pattern "^[\\w._%+-]+@[\\w.-]+\\.[a-zA-Z]{2,}$" ;  # 이메일 형식
        sh:message "올바른 이메일 주소를 입력하세요."@ko
    ] .
```

## 핵심 제약 구성요소 (Constraint Components)

### 카디널리티 제약

```turtle
sh:property [
    sh:path :hasChild ;
    sh:minCount 0 ;    # 최소 0개 (없어도 됨)
    sh:maxCount 10 ;   # 최대 10개
] .
```

### 값 타입 제약

```turtle
sh:property [
    sh:path :birthDate ;
    sh:datatype xsd:date ;         # 날짜 타입이어야 함
] ;
sh:property [
    sh:path :salary ;
    sh:datatype xsd:decimal ;
    sh:minInclusive 0 ;            # 0 이상
    sh:maxInclusive 1000000000 ;   # 10억 이하
] .
```

### 값 범위 제약

```turtle
sh:property [
    sh:path :age ;
    sh:minInclusive 0 ;
    sh:maxExclusive 150 ;
] .

sh:property [
    sh:path :riskScore ;
    sh:minExclusive 0.0 ;    # 0 초과 (0은 포함 안 됨)
    sh:maxInclusive 1.0 ;    # 1 이하
] .
```

### 패턴 제약

```turtle
sh:property [
    sh:path :phoneNumber ;
    sh:pattern "^\\+?[0-9]{10,15}$" ;    # 국제 전화번호 형식
    sh:flags "i" ;                         # 대소문자 무시
] ;
sh:property [
    sh:path :employeeId ;
    sh:pattern "^EMP-[0-9]{6}$" ;         # EMP-000001 형식
    sh:message "직원 ID는 EMP-숫자6자리 형식이어야 합니다."@ko
] .
```

### 클래스 제약

```turtle
sh:property [
    sh:path :worksAt ;
    sh:class :Organization ;    # worksAt의 목적어는 Organization이어야 함
    sh:minCount 1 ;
] ;
sh:property [
    sh:path :hasPrescription ;
    sh:node :PrescriptionShape ;  # 중첩 Shape 참조
] .
```

### 열거 제약

```turtle
sh:property [
    sh:path :bloodType ;
    sh:in ( :A :B :AB :O ) ;     # 허용 값 목록
] ;
sh:property [
    sh:path :status ;
    sh:in ( "active"^^xsd:string
            "inactive"^^xsd:string
            "pending"^^xsd:string ) ;
] .
```

### SPARQL 기반 제약 (고급)

복잡한 제약은 SPARQL 쿼리로 표현할 수 있습니다.

```turtle
:PrescriptionShape
    rdf:type sh:NodeShape ;
    sh:targetClass :Prescription ;
    sh:sparql [
        sh:message "처방 날짜는 환자 입원 날짜 이후여야 합니다."@ko ;
        sh:prefixes :prefixDecl ;
        sh:select """
            SELECT $this WHERE {
                $this :prescriptionDate ?pDate ;
                      :forPatient/:admissionDate ?aDate .
                FILTER (?pDate < ?aDate)
            }
        """ ;
    ] .
```

## 검증 보고서 (ValidationReport)

SHACL 검증 엔진(예: TopBraid SHACL, pySHACL, Apache Jena SHACL)은 검증 결과를 **ValidationReport** 형식의 RDF로 반환합니다.

```turtle
# 검증 실행 결과 예시
[] rdf:type sh:ValidationReport ;
   sh:conforms false ;              # 검증 실패
   sh:result [
       rdf:type sh:ValidationResult ;
       sh:resultSeverity sh:Violation ;      # 심각도: Violation
       sh:focusNode :직원홍길동 ;              # 문제 발생 노드
       sh:resultPath :hasEmail ;             # 문제 속성
       sh:resultMessage "올바른 이메일 주소를 입력하세요."@ko ;
       sh:sourceConstraintComponent sh:PatternConstraintComponent ;
       sh:sourceShape :PersonShape
   ] , [
       rdf:type sh:ValidationResult ;
       sh:resultSeverity sh:Warning ;        # 심각도: Warning
       sh:focusNode :직원이영희 ;
       sh:resultPath :hasName ;
       sh:resultMessage "이름이 없습니다."@ko ;
       sh:sourceConstraintComponent sh:MinCountConstraintComponent
   ] .
```

### 심각도 수준

| 심각도 | 의미 | 처리 방식 |
|--------|------|---------|
| `sh:Violation` | 위반 — 데이터가 제약을 어김 | 파이프라인 중단 또는 격리 |
| `sh:Warning` | 경고 — 데이터 품질 이슈 | 로그 기록, 검토 필요 |
| `sh:Info` | 정보 — 주의 사항 | 알림만 |

## SHACL과 OWL의 상호 보완

| 역할 | OWL (추론) | SHACL (검증) |
|------|-----------|-------------|
| 세계 가정 | OWA | CWA 스타일 |
| 목적 | 새 지식 도출 | 데이터 품질 검증 |
| 위반 처리 | 불일관성 (논리적) | 보고서 생성 (운영적) |
| 언제 실행 | 온톨로지 로드 시 | 데이터 수집·업데이트 시 |
| 표준 | W3C OWL 2 | W3C SHACL |

실무 아키텍처에서 두 가지를 함께 사용합니다.

1. **OWL** — 온톨로지 TBox 정의, 추론 (분류, Subsumption, Consistency)
2. **SHACL** — ABox 데이터 입력 시 품질 검증, 운영 제약 적용

```
데이터 수집 → SHACL 검증 → [통과] → RDF 저장소 → OWL 추론 → 결과 활용
                         → [실패] → 검증 보고서 → 데이터 수정
```

## pySHACL 예시 (Python)

```python
from pyshacl import validate
import rdflib

# 데이터 그래프
data_graph = rdflib.Graph()
data_graph.parse("employees.ttl", format="turtle")

# SHACL 형태 그래프
shapes_graph = rdflib.Graph()
shapes_graph.parse("employee-shapes.ttl", format="turtle")

# 검증 실행
conforms, results_graph, results_text = validate(
    data_graph,
    shacl_graph=shapes_graph,
    inference='rdfs',     # RDFS 추론 후 검증
    abort_on_first=False  # 모든 위반 수집
)

print(f"검증 통과: {conforms}")
if not conforms:
    print(results_text)
```

다음 챕터에서는 시소러스와 통제 어휘를 RDF로 표현하는 표준인 SKOS를 학습합니다.
