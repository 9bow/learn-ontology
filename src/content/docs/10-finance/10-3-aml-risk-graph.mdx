## 자금세탁의 구조와 그래프의 필요성

자금세탁은 본질적으로 **관계의 조작**입니다. 불법 자금은 직접 이동하지 않고 여러 법인·계좌·국가를 거치며 출처를 가립니다. 전통적인 규칙 기반 AML 시스템은 개별 거래의 임계값(예: 1만 달러 이상 현금 거래)을 감시하지만, 복잡하게 분산된 거래 패턴은 탐지하기 어렵습니다.

지식 그래프 기반 AML은 다르게 접근합니다. 계좌, 법인, 개인, 거래를 노드로, 자금 이동·소유 관계·주소 공유 등을 엣지로 표현합니다. 그러면 "A 계좌 → B 법인 → C 계좌 → D 개인 → A 계좌"처럼 순환하는 자금 흐름 패턴을 그래프 알고리즘으로 탐지할 수 있습니다.

## 그래프 기반 AML의 핵심 패턴

실제 AML 분석에서 자주 등장하는 패턴들입니다.

**레이어링(Layering):** 자금을 여러 중간 계좌를 통해 이동시켜 출처를 숨기는 방식입니다. 그래프에서는 특정 기간 내에 생성된 계좌들이 단방향으로 자금을 전달하는 체인 구조로 나타납니다.

**셸 컴퍼니 구조:** 여러 국가에 걸쳐 설립된 법인들이 소유 관계로 연결되어 실제 수익 소유자(Ultimate Beneficial Owner, UBO)를 숨기는 구조입니다. EU의 5차 자금세탁방지 지침(5AMLD)은 UBO 정보를 공개 등록부에 기재하도록 의무화했으며, GLEIF의 LEI 데이터와 연계하면 이 구조를 그래프로 표현할 수 있습니다.

**공유 속성 클러스터링:** 서로 다른 계좌가 같은 전화번호, IP 주소, 주소를 공유하는 경우를 엣지로 연결하면 실제로는 한 사람이 통제하는 여러 계좌 클러스터가 드러납니다.

```sparql
# 특정 법인의 UBO를 최대 5단계 소유 체인에서 찾는 SPARQL 예시
SELECT ?ulo ?person WHERE {
  :TargetCompany fibo-be:isOwnedBy+ ?ulo .
  ?ulo a fibo-be:NaturalPerson .
  ?ulo fibo-be:hasOwnershipPercentage ?pct .
  FILTER(?pct >= 0.25)
}
```

## 자금세탁 그래프 패턴

```text
Smurfing (분산 입금) 패턴
──────────────────────────────────────────────────────────
  불법 자금
     │
     ├──▶ [개인 A] ──$9,800──▶ [계좌 X1]
     ├──▶ [개인 B] ──$9,500──▶ [계좌 X2]  ← 각각 $10,000
     ├──▶ [개인 C] ──$9,700──▶ [계좌 X3]    미만으로 분산
     └──▶ [개인 D] ──$9,200──▶ [계좌 X4]
                                    │
                               집결 이체
                                    ▼
                              [계좌 Y] (자금 세탁 완료)

Layering (다단계 레이어링) 패턴
──────────────────────────────────────────────────────────
  [계좌 A]                              [최종 계좌]
  한국법인  ──▶ [Shell Co. 케이맨] ──▶  [신탁 BVI]  ──▶ 합법 자산
     ▲              │                       │
     │         소유 관계               소유 관계
     │         (25% 미만)             (25% 미만)
  실제 수익             UBO(실제 소유자)가 숨겨짐
  소유자

  그래프 탐색: :TargetCompany fibo-be:isOwnedBy+ ?ulo
               → 모든 중간 법인 경유, UBO 탐지
```

## 규제 보고와 BCBS 239

금융 규제 보고는 단순히 숫자를 제출하는 것이 아닙니다. **"이 숫자가 어디서 왔는가"를 설명할 수 있어야 합니다.** 바젤위원회의 BCBS 239 원칙(2013)은 이를 명시적으로 요구합니다.

BCBS 239의 14개 원칙 중 특히 데이터 계보와 관련된 핵심 원칙들입니다.

| 원칙 | 요약 |
|------|------|
| 원칙 3: 정확성과 무결성 | 리스크 데이터는 정확해야 하며, 자동화된 조정(reconciliation) 절차가 있어야 한다 |
| 원칙 4: 완전성 | 모든 중요 리스크 데이터를 포괄해야 한다 |
| 원칙 6: 적시성 | 정상 상황에서 주별, 위기 상황에서 일별 보고가 가능해야 한다 |

온톨로지와 지식 그래프는 BCBS 239 준수를 위해 다음을 제공합니다.

- **데이터 계보(Data Lineage):** 각 리스크 수치 트리플에 출처 시스템, 변환 프로세스, 타임스탬프를 Named Graph로 부착합니다.
- **어휘 표준화:** FIBO를 참조해 "시장 리스크", "신용 리스크" 같은 개념을 기관 전체에서 동일하게 정의합니다.
- **자동 조정:** SHACL 제약 조건으로 리스크 데이터 간 불일치(예: 부서별 익스포저 합이 전사 집계와 불일치)를 자동 탐지합니다.

## 리스크 집계와 그래프 순회

리스크 집계에서 그래프가 특히 유용한 이유는 **계층적 소유 구조**를 처리할 때입니다. 예를 들어 은행 그룹이 100개 자회사를 가지고 있고, 각 자회사가 또 다른 자회사를 가진다면, 전체 그룹의 신용 리스크 익스포저를 계산하려면 소유 구조 트리를 순회해야 합니다.

관계형 데이터베이스에서는 재귀 쿼리(Recursive CTE)로 이를 처리하지만, 소유 구조가 트리가 아닌 DAG(방향 비순환 그래프) 형태일 경우 복잡해집니다. 그래프 데이터베이스는 이를 자연스럽게 처리합니다.

```sparql
# 특정 법인과 그 모든 자회사의 총 신용 익스포저 합산
SELECT ?entity (SUM(?exposure) AS ?totalExposure) WHERE {
  :ParentBank fibo-be:isOwnedBy* ?entity .
  ?entity :hasCreditExposure ?exposure .
}
GROUP BY ?entity
```

## 실시간 모니터링과 그래프 업데이트

AML과 리스크 모니터링은 배치 처리만으로는 부족합니다. 의심 거래는 실시간으로 탐지해야 합니다. 이를 위해 다음 아키텍처가 사용됩니다.

**스트리밍 그래프 업데이트:** Kafka 같은 메시지 스트림에서 거래 이벤트를 수신하고, 그래프 데이터베이스에 실시간으로 엣지를 추가합니다.

**증분 패턴 탐지:** 새 엣지가 추가될 때마다 관련 패턴(예: 순환 자금 흐름)을 증분적으로 검사합니다. 전체 그래프를 매번 재분석하는 대신, 변경된 부분 주변만 검사합니다.

**임계값 기반 경보:** 특정 패턴이 탐지되면 Suspicious Activity Report(SAR) 초안을 자동 생성하고 담당자에게 알립니다.

## 프라이버시와 규제 준수의 균형

AML 그래프는 방대한 개인 금융 정보를 포함합니다. GDPR(유럽)이나 한국의 개인정보 보호법 준수를 위해 다음 사항을 설계 단계부터 고려해야 합니다.

- **데이터 최소화:** AML 목적에 필요한 최소한의 개인 식별 정보만 그래프에 포함합니다.
- **접근 제어:** AML 분석팀과 일반 비즈니스팀 간에 접근 가능한 그래프 데이터를 분리합니다.
- **삭제권 대응:** GDPR의 "잊혀질 권리" 요청 시 관련 개인 데이터를 식별하고 삭제할 수 있는 메커니즘이 필요합니다.

## 정리

지식 그래프는 AML 탐지에서 규칙 기반 시스템이 놓치는 복잡한 관계 패턴을 발견하고, BCBS 239 같은 규제 요건을 충족하는 데이터 계보를 제공합니다. 다음 챕터에서는 ESG 데이터와 자산 관리에서의 온톨로지 적용을 살펴봅니다.
